# Research: Customer / Loyalty Grain + Dimension Pattern (098)

Phase 0 -- precedent survey, input-source confirmation, and the deferred-capability
note. Every read below was of a committed artifact in this worktree; no live DB, no
Power BI/PBIP surface, no network was touched.

## Precedents (what SHIPPED to reuse, what to stay distinct from)

- **`gold.dim_customer_rss` + `gold.fct_sales_rss`**
  (`warehouse/migrations/0004_create_gold_retail_store_sales_star.sql`; mirrored in
  `powerbi/RetailStoreSales.SemanticModel/definition/tables/gold dim_customer_rss.tmdl`).
  SHIPPED. This is the ONE filled, table-scoped instance of a customer dimension in the
  repo: `customer_sk INT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`, a bare
  `customer_id TEXT` natural key, a `(-1, 'UNKNOWN')` unknown-member row, FK
  `COALESCE(dc.customer_sk, -1)` on the fact -- i.e. RC14 applied once, on one already-
  answered PII ruling (`mappings/retail_store_sales/unresolved-questions.md` Q1: "KEEP
  `customer_id` -- pseudonymous surrogate, no raw PII"). REUSE from this instance: the
  RC14 mechanics themselves (surrogate `_sk`, `-1` unknown member, FK COALESCE) as the
  dimension pattern's structural convention -- this feature invents no new key
  convention (spec Assumptions). STAY DISTINCT: this feature does NOT edit the 0004
  migration or its TMDL mirror (FR-007), does NOT re-open Q1, and does NOT generalize
  Q1's answer ("keep, no raw PII") into a repo-wide or pattern-level default -- the new
  pattern's identity-key and PII-publish slots stay unfilled placeholders, never
  `customer_id` or "keep" baked in as the shown example (Principle VII). `dim_customer_rss`
  is also structurally thinner than what this feature must define: it has no
  identity-resolution shape (one row already equals one already-trusted natural key;
  nothing here resolves the same person appearing under multiple raw ids) and no
  SCD/historization marker (silently Type 1 by construction, never named as a choice).
  Those two gaps are exactly what User Story 3 and FR-002's new SCD slot supply
  generically.

- **`skills/retail-kpi-knowledge/domains/customer.md`** (spec 042). SHIPPED. This is the
  KNOWLEDGE-LAYER neighbour this feature cites, never restates. It lists four Planned
  KPIs (Customer Retention Rate, Purchase Frequency, Customer Lifetime Value,
  New-vs-Returning split) and states four Principle-V stops verbatim (identity/grain,
  PII publish-safety, business-segment rollups, product identity), deciding none.
  REUSE: the four KPI names (FR-003 must produce one candidate-grain entry per KPI) and
  the four stop names (FR-006's cross-reference target) verbatim, so a reader can trace
  from this feature's structural pattern back to the exact ambiguity the domain file
  already named. STAY DISTINCT: this feature does not edit `domains/customer.md`'s KPI
  table, decision-questions table, or Principle-V stop section (FR-008); it adds a
  STRUCTURE-layer companion doc set, never a second copy of the knowledge layer.

- **`templates/source-map.yaml`** (spec 001, Principle IV). SHIPPED, actively used by
  every table's mapping gate. This is the CONSUMER surface FR-014 targets: a future
  table's `gold_star.dimensions[]` entry should be able to cite the new customer
  dimension template BY NAME (e.g. `name: "dim_customer"`, referencing
  `templates/customer-dimension.md` for the slot shape) without the template's prose
  being copied into the map. REUSE: the existing `gold_star.dimensions[].name` /
  `.surrogate_key` / `.has_unknown_member` / `.attributes[]` field shape as the target
  a filled customer dimension instance would populate -- the new template's slots map
  onto these same fields, so a table that adopts the pattern still produces an ordinary
  `source-map.yaml` entry, not a new machine-readable format. STAY DISTINCT: this
  feature adds NO new key to `source-map.yaml`'s schema (that surface is shared across
  many in-flight features and is outside this feature's collision-avoidance
  allocation) -- it only produces citable prose/template content the map can point at
  by name.

- **`docs/decisions/0002-retail-cleaning-defaults.md`** (RC14, RC15). SHIPPED. RC14
  ("Gold is a Kimball star: one fact at the silver grain + conformed dims; surrogate
  `_sk` keys; an unknown member at `_sk = -1` per dim with fact FKs COALESCE'd to -1")
  is the exact mechanical convention both new pattern documents cite for their
  surrogate-key and FK-join sections (FR-002, FR-003). REUSE verbatim; this feature does
  not invent a competing key convention (spec Assumptions).

- **`.specify/templates/plan-template.md`** and the **087-conformed-dimension-readiness**
  slice (`specs/087-conformed-dimension-readiness/*`). SHIPPED, most recent
  precedent for this Plan stage's document shape (Constitution Check table format,
  research.md precedent-survey structure, data-model.md entity-table format,
  quickstart.md numbered-walkthrough format). REUSE the DOCUMENT SHAPE only. STAY
  DISTINCT at the CONTENT layer: 087 adds a new `retail check` rule (HR1) and touches
  `src/retail/rules/`, the wiring meta-gate, `EXPECTED_RULE_IDS`, `rules-manifest.json`,
  `severity-posture.json`, and the glossary rule-count anchor. This feature (098) adds
  NO static rule and touches NONE of those wiring surfaces (collision-avoidance

## Deferred capabilities NOT assumed

- **F016 (Power BI execution adapter)** (official Power BI MCP / connection; `pbi-cli`
  no longer preferred) is gated + LAST and is assumed NOT to exist. Neither new pattern
  document nor the template invokes it, references a live publish action, or assumes a
  materialized semantic model.
- **No live database connection.** This feature performs no `retail validate` run, no
  Postgres query, and no live profiling of any real customer table. Pattern authorship
  is static-only (Principle VIII); any live application of the pattern to a real table
  is a future table's own Stage 1/2 profiling work, explicitly out of scope here
  (FR-013).
- **No metric contract seeding.** F009's contract-template + review process is
  untouched; no file under `contracts/` is created or modified (FR-009, SC-007).
- **No identity-resolution algorithm, merge heuristic, or precedence rule.** Identity
  resolution across multiple raw customer ids is named as a reserved owner ruling only
  (FR-005, User Story 3); no matching logic is authored or implied.
- **No `retail check` rule.** This feature adds zero rule ids and touches none of the
  static-checker wiring surfaces (`src/retail/rules/`, the wiring meta-gate,
  `EXPECTED_RULE_IDS`, `rules-manifest.json`, `severity-posture.json`, the glossary
  rule-count anchor) -- collision-avoidance allocation, FR-001, FR-015.
- **No roadmap F-number allocation.** Per the spec's own Assumptions, this spec does not
  require a new roadmap F-number decision; none is invented in this plan stage.
- **No PII, grain, retention-window, CLV-horizon, or new-vs-returning-anchor ruling.**
  All five (plus the SCD Type-1-vs-2 choice the clarify session added) are named as
  explicit, unresolved owner rulings across the two new documents and the template, per
  Principle V -- see the "Open (Principle V)" section below.

## Open (Principle V -- NOT resolved here; carried to a future table's owner)

These are not decided by this feature and are not decided by this plan stage. They are
listed here so the design set is explicit about what stays open, matching spec.md's
Clarifications Q5:

  merged into one `customer_sk`) -- FR-005, User Story 3.
- PII publish-safety for a future table's customer identifier(s) -- FR-002, FR-006.
- SCD/historization type (Type 1 overwrite vs. Type 2 track-history) for the customer
  dimension -- FR-002 (Clarify Q4).
- Retention-window definition -- FR-004.
- CLV horizon and discounting choice -- FR-004.
- New-vs-returning first-purchase anchor -- FR-004.

Each stays exactly as `domains/customer.md` and spec.md's FR-004/FR-005/FR-006 already
frame it: a reserved judgment call for a real table's named business/governance owner,
not decided in this feature and not decided by this planning stage.
