# Feature Specification: Contract-Driven Discovery-to-Decision Flow

**Feature Branch**: `122-contract-driven-discovery`

**Created**: 2026-07-12

**Status**: Draft

**Input**: User description: "Close the first major product gap between a real multi-table retail data source and governed, human-approved business decisions. Let an agent safely (1) inspect a real database or supported file source read-only; (2) produce a reviewable multi-table discovery profile; (3) propose a retail domain and a bounded first-project scope; (4) hand the grounded discovery evidence into the already-shipped Business Knowledge Interview and Decision Store; (5) use the existing Database-to-PBIP Knowledge Contracts to determine the one next allowed action; (6) stop truthfully whenever required evidence, decisions, or human approvals are missing. Product specification only."

## Overview

### Feature Summary

Spec 121 shipped the **decision foundation** of the Database-to-PBIP flow: the machine-readable Decision Store, the Business Knowledge Interview behavior, the per-stage Knowledge Contracts (`contracts/knowledge/database-to-pbip-flow.yaml`), the approval-authority map, and the pass/warn/blocked gate verdicts. Those contracts already *declare* three early stages — `discovery`, `domain_guess`, `scope_proposal` — each with `required_inputs`, `required_outputs`, stop rules, and a handoff target. But 121 explicitly listed **"Automating discovery, domain guessing, or scope proposal"** as out of scope ("Future Slice 1"). Nothing on `main` today **produces** the artifacts those three contracts promise: there is no multi-table discovery profile, no recorded domain-guess decision, and no recorded scope proposal. The interview therefore has a declared input it cannot yet be fed by the tool.

This feature builds the **behavior** for those already-declared contracts. Given a reachable real retail source, an agent performs read-only discovery in two layers — a portfolio-level metadata survey (Layer A) across all reachable tables, then deep value-backed profiling (Layer B) only for the tables selected into the proposed scope, through the existing per-table profiler — proposes a retail domain and a bounded first scope as **proposed** (not approved) decisions in the existing Decision Store, and hands the grounded evidence into the already-shipped Business Knowledge Interview. At every step the agent consults the existing Knowledge Contracts to determine the single next allowed action **within this bounded discovery-to-handoff flow**, and stops truthfully — naming the concrete local missing artifact, decision, or approval — whenever evidence or a human decision is absent. It does not repair or extend the general cross-stage Decision Gate. It writes no Silver/Gold SQL, defines no KPI meaning, generates no DAX/PBIP, and grants no approval.

### Problem Statement

A new external user can install the kit and reach the shipped decision layer, but there is no governed on-ramp **into** it from a real, multi-table source. The single-table front door (`retail-onboard-table`) walks *one* `<schema>.<table>` from Source Ready to Mapping Ready; it does not survey a whole database, propose which subject area the data represents, or scope a first delivery slice. Confronted with a source of dozens or hundreds of tables, the agent today has no governed way to answer "what is this, and where do we start?" without inventing meaning. The consequence is either paralysis (the user must read the whole repo and drive table-by-table) or ungoverned guessing (the agent silently decides the domain and scope). This feature closes that gap while preserving every existing gate: discovery is evidence, not meaning; domain and scope are proposals, not approvals; the interview and Decision Store are reused, not reimplemented.

### Product Goal

A BI developer, data analyst, retail operations analyst, consultant, or agent operator can point Seshat BI at a real retail source and receive a **governed, reviewable path from source discovery to recorded business decisions** — a committed multi-table portfolio survey (Layer A), deep per-table profiles for the in-scope tables via the existing profiler (Layer B), a proposed domain, a proposed first scope, and a clean handoff into the existing interview — **without** the agent inventing meaning, executing transformations, exposing raw PII, bypassing readiness gates, or granting approval. At any point the agent can name exactly one next allowed action from committed evidence and the existing contracts, and it stops truthfully when required evidence, decisions, or approvals are missing.

### Primary Actors

- **BI developer / data analyst** who supplies the source and drives the agent.
- **Data owner / business owner** who confirms business meaning and approves critical decisions (never the agent).
- **Agent operator** using Claude Code, Codex, or another compatible agent (the agent is the runtime; Principle I).
- **Human reviewer / approver** with an eligible authority class per `contracts/knowledge/approval-authority.yaml`.
- **Seshat governance layer** — the readiness spine, the Knowledge Contracts, the Decision Store, and the static gate — which evaluates committed evidence and blocks unsafe progression. It owns pass/block; it never invents answers.

## Clarifications

### Session 2026-07-12

Three product decisions that scoped User Stories 2–4 were resolved by the owner (recommended option in each case):

- Q: How do domain/scope proposals enter the store without a second vocabulary (121 has no `domain_classification`/`scope_proposal` type)? → A: **Non-critical proposals, confirmed through the existing low-risk (batch) interview confirmation path** — no new critical `decision_type`, no new `approval-authority.yaml` row, no new status; 121's vocabulary, statuses, and authority map stay frozen (FR-019). Lifecycle uses only existing statuses and record fields: the agent records a `proposed` non-critical record (with `confidence`); a named human confirms it as a member of a `batches[]` entry (`confirmed_by` = named human + authority class) — the low-risk batch path, which for a non-critical type triggers NO authority-class eligibility check and needs NO new authority-map row; a refusal is `status: rejected`; a change is a supersession (`supersedes`/`superseded_by`). There is NO `confirmed` status and NO new confirmation mechanic — batch confirmation of a non-critical type is exactly 121's existing low-risk path, and the exact recorded status of a confirmed member follows 121's convention.
- Q: What is the delivery surface under the ratified Option-B (skill-driven) direction? → A: **A new dedicated skill**, mirroring how 121 shipped `business-knowledge-interview` (FR-005).
- Q: How does the multi-table profile relate to the existing single-table `source-profile.md`? → A: **Portfolio survey first, then per-table onboarding** — a new portfolio-level survey artifact precedes the existing `retail-onboard-table` per-table profiler (FR-013). This resolves into **two explicit layers** (FR-009/FR-013): **Layer A (portfolio survey)** reads only read-only source *metadata* and bounded, non-value observations across every reachable table (schema/table identity, column inventory, declared types, declared PK/FK metadata where exposed, approximate/metadata row count where safe, name-and-type-based date/PII *hints*, structural role hints, coverage limits, candidate domain evidence, candidate first-scope tables) — observations and hints, never approved semantics; **Layer B (deep per-table profile)** runs value-backed checks (actual missingness, blank-vs-null, measured PK uniqueness/nullability, value-backed grain and relationship/orphan evidence, measured date coverage, masked value samples, returns-column population) ONLY for tables selected into the proposed first scope, through the existing per-table onboarding/profiler — the new feature composes/invokes that profiler and never re-implements `mappings/<table>/source-profile.md`.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Produce a Reviewable Multi-Table Portfolio Survey (Priority: P1) — MVP

A user supplies a reachable database or a supported file source (or a folder of files). The agent performs a **read-only, portfolio-level survey (Layer A)** across all reachable tables using only source *metadata* and bounded, non-value observations, and produces one committed, reviewable multi-table survey artifact. The survey is *evidence and hints* — metadata facts and honestly-marked unknowns — never approved business meaning, and it does NOT perform deep value-backed profiling of every reachable table. Deep value-backed profiling (Layer B) runs later, and only for the tables selected into the proposed first scope, through the existing per-table onboarding/profiler (see FR-013, User Story 3).

**Why this priority**: The portfolio survey is the single input that the domain guess, the scope proposal, and the Business Knowledge Interview all depend on. It is independently valuable (a user gets a legible orientation across an unfamiliar source without reading the repo) and independently testable. It is the smallest slice that delivers the feature's core promise, so it is the MVP. Because deep per-table profiling is confined to in-scope tables (a downstream, P2 step), no deep profiling occurs at MVP — the MVP is the Layer-A survey only.

**Independent Test**: Point the flow at a worked-example multi-table source (or a fixture of table/column *metadata*). Verify a single committed portfolio-survey artifact is produced covering every reachable table from metadata alone; that each metadata fact is recorded and every unavailable metadata item is marked with its exact reason; that date/PII *hints* are recorded as name-and-type-based hints (not measured rulings); that the artifact contains no value-backed measurement (no measured uniqueness, no measured missingness, no date-coverage measurement, no raw samples) that would duplicate the per-table profiler; and that no raw PII, credential, DSN, or connection string appears in the committed artifact.

**Acceptance Scenarios**:

1. **Given** a reachable multi-table source, **When** the Layer-A portfolio survey runs read-only, **Then** the committed survey records, per reachable table, only read-only metadata and bounded observations: source/table identity (without credentials), table and column inventory, declared data types, declared PK/FK metadata where the source exposes it, candidate grain inferred from that declared metadata (the `discovery` contract's "candidate grains" output, at the metadata level — not a value-backed proof), approximate/metadata row count or size where safely available, column-name-and-type-based date hints, column-name-and-type-based PII suspicion hints, possible structural role hints (transaction / snapshot / dimension / reference / technical), stated coverage limits and unavailable metadata, candidate domain evidence, and candidate tables for the first scope — recorded as observations and hints, never as approved semantics.
2. **Given** a column whose name or type raises a PII suspicion hint, **When** the survey records it, **Then** it is recorded as a suspicion *hint* only (no value is sampled at Layer A) and no raw suspected-PII value is written to the committed artifact.
3. **Given** a source with hundreds of tables, **When** the survey runs, **Then** it inventories all reachable tables from metadata and, if any coverage limit is applied, the limit and what was omitted are stated explicitly (never a silent truncation presented as complete).
4. **Given** metadata access is unavailable, an optional reader is missing, or a metadata item is unavailable, **When** the survey runs, **Then** the affected observations are marked `[PENDING LIVE PROFILE]` / `needs_sample` with the boundary named and the enabling step stated; the artifact records `warning`/`blocked` semantics as appropriate and never a fabricated value or false pass.
5. **Given** the survey is rerun after new tables or columns appear, **When** the survey regenerates, **Then** newly observed structure is reflected and no previously committed decision is overwritten (drift is surfaced, not silently absorbed).
6. **Given** a candidate primary key visible in declared PK/FK metadata, **When** the survey records it, **Then** it is recorded as a *candidate from declared metadata* only; the survey does NOT measure uniqueness or nullability on the rows — value-backed PK proof is a Layer-B check that runs through the existing per-table profiler when the table is selected into scope (User Story 3, FR-013).
7. **Given** a table selected into the proposed first scope, **When** deep per-table profiling (Layer B) is needed (actual missingness, blank-vs-null, measured PK uniqueness/nullability, value-backed grain and relationship/orphan evidence, measured date coverage, masked value samples, returns-column population), **Then** the feature hands that table to the existing per-table onboarding/profiler (`retail-onboard-table` / Source Ready) and does NOT perform those value-backed checks in the portfolio survey or re-author `mappings/<table>/source-profile.md`. (This scenario belongs to the scope step, User Story 3 / P2 — it does not run at MVP.)

### User Story 2 - Propose a Retail Domain from Discovery Evidence (Priority: P2)

From the committed discovery evidence (the Layer-A portfolio survey), the agent proposes a retail domain or subject area (for example sales, inventory, purchasing, customers, products, or finance). The proposal is recorded as a `proposed` decision in the existing Decision Store, citing the specific survey facts it rests on. The agent never silently approves the classification.

**Why this priority**: The domain guess narrows the interview and the scope proposal, but it depends entirely on User Story 1's survey and can be delivered after it. It is a proposal, so its value is realized only once a named human confirms it (via the low-risk batch path), rejects it, or supersedes it.

**Independent Test**: Run the domain guess against a committed portfolio-survey fixture. Verify the agent records a domain proposal in the Decision Store with `status: proposed` and a `confidence` value, cited evidence (the survey facts), `proposed_by` = the agent (never an approval); verify a named-human acceptance is recorded as a `batches[]` confirmation (with `confirmed_by` = named human + authority class, `batch_id` on the member) on the non-critical batch path (no authority-class eligibility check, no new authority-map row), that the agent never self-confirms, that a rejection is recorded as `status: rejected`, and that a change is recorded via supersession; verify an ambiguous source yields either an explicitly-uncertain proposal or a recorded "cannot determine domain from available evidence" outcome — never a confident guess dressed as fact.

**Acceptance Scenarios**:

1. **Given** a committed portfolio survey, **When** the domain guess runs, **Then** the agent records a domain proposal in the existing Decision Store as a `proposed` decision (with `confidence`, `proposed_by` = the agent) citing the survey evidence; it stays agent-proposed until a named human acts — confirmation is a named-human `batches[]` entry (`confirmed_by` = named human + authority class) on the low-risk batch path (non-critical, no authority-class eligibility check, no new authority-map row), a refusal makes it `rejected`, and a change supersedes it. The agent never self-confirms.
2. **Given** the domain is ambiguous (evidence points to multiple domains or none), **When** the guess runs, **Then** the ambiguity is recorded explicitly (alternatives with their competing evidence, or an honest "undetermined") and no single domain is asserted.
3. **Given** a recorded domain proposal, **When** any consumer reads it, **Then** the agent's confidence value is visible and is never presented as readiness or approval.
4. **Given** no committed discovery profile exists, **When** the domain guess is requested, **Then** the contract's stop rule halts it and names the missing profile as the unblocking artifact.

### User Story 3 - Propose a Bounded First-Delivery Scope (Priority: P2)

The agent proposes a bounded first scope using deterministic, evidence-based bounding: candidate tables, candidate business questions, candidate KPIs (named as candidates only — not defined), explicitly excluded tables or domains, unresolved dependencies, and the decisions required from the owner. The proposal honors an explicit user-supplied scope limit when one exists; otherwise it prefers one coherent business process or subject area, one primary fact grain, and candidate KPIs that can share a coherent model boundary. It never invents a size judgment, a numeric scope score, a hidden table-count threshold, or a priority rank, and it never claims one scope is objectively correct from structural evidence alone. When the evidence crosses multiple business processes, fact grains, unrelated owner groups, or incompatible model boundaries, the proposal records the scope categorically as **cross-boundary** and either presents narrower coherent candidate options or records `needs_user_input` for the human choice. The proposal is agent-authored as `proposed`; a named human confirms it (via the low-risk batch path), rejects it, supersedes it, or accepts only part of it (a bounded superseding proposal) — see the FR-019 lifecycle. The agent never confirms it itself.

**Why this priority**: Scope depends on both the profile and the domain guess. A coherent first scope is what makes the subsequent interview focused (fewer rounds than the in-scope column count), but it is a proposal that a human must confirm, so it follows domain guess.

**Independent Test**: Run the scope proposal against a committed portfolio survey and a recorded domain proposal. Verify the agent records a scope proposal (`status: proposed`, `proposed_by` = the agent) with candidate tables, candidate questions, candidate KPI *names*, explicit exclusions, unresolved dependencies, and required owner decisions; verify it is not self-confirmed by the agent and that a named-human `batches[]` confirmation on the non-critical batch path is what accepts it (no authority-class eligibility check, no new authority-map row); verify an explicit user-supplied scope limit is honored when present; verify that when the evidence supports one coherent business process/fact grain the scope is described categorically as **coherent**, and when it crosses multiple processes/grains/owner groups/model boundaries the proposal is described categorically as **cross-boundary** and presents narrower coherent options or records `needs_user_input` — with no numeric score, table-count threshold, fabricated rank, or silent default anywhere.

**Acceptance Scenarios**:

1. **Given** a committed portfolio survey and a recorded domain proposal, **When** the scope proposal runs, **Then** it records candidate tables, candidate business questions, candidate KPI names, explicitly excluded tables/domains, unresolved dependencies, and the decisions the owner must make — all as a `proposed` decision (with `confidence`) citing survey + domain evidence — preferring one coherent business process/subject area, one primary fact grain, and KPIs that share a coherent model boundary, and honoring any explicit user-supplied scope limit.
2. **Given** the scope evidence crosses multiple business processes, fact grains, unrelated owner groups, or incompatible model boundaries, **When** the proposal is composed, **Then** it is recorded categorically as **cross-boundary**, presents narrower coherent candidate options where the evidence supports more than one coherent slice, and records `needs_user_input` when a human choice is required; it is not silently enlarged, and no numeric score, hidden table-count threshold, fabricated priority rank, or silent default is used.
3. **Given** a candidate KPI appears in scope, **When** it is recorded, **Then** it is named as a *candidate KPI* only; its business meaning is NOT defined here and any meaning question routes to the Retail KPI knowledge boundary.
4. **Given** no recorded domain proposal exists, **When** the scope proposal is requested, **Then** the `scope_proposal` contract's **stop rule / `required_inputs`** halts it and names the missing domain-guess decision. (This sequencing gate is enforced by the stage contract's `required_inputs`/`stop_rules` — NOT by `blocking_decision_categories`; consistent with FR-019, a domain/scope decision_type never sits in a blocking category, yet a missing upstream proposal still halts the next stage via the contract's input requirement.)
5. **Given** a named human accepts only part of a proposed scope, **When** the acceptance is recorded, **Then** a NEW bounded `proposed` scope record covering only the accepted subset is created that `supersedes` the original, and the original is marked `superseded` (with `superseded_by` set) — the original is never silently mutated, and no two active records of the same type share a scope key (FR-019B).

### User Story 4 - Hand Grounded Discovery into the Business Knowledge Interview (Priority: P3)

The feature passes the committed discovery profile and the proposed scope into the already-shipped Business Knowledge Interview, matching the interview contract's declared `required_inputs`. The interview's declared "committed discovery profile" input is the Stage-1 per-table (Layer-B) profile of the in-scope tables (the interview contract names it "Stage-1 read-only profile of `retail-onboard-table` or equivalent"); the Layer-A portfolio survey orients the run and feeds the domain/scope proposals but is not a substitute for the per-table profile the interview requires. The existing Decision Store is loaded first; existing decisions are presented for confirmation or supersession, never overwritten. Interview questions are grounded in discovery evidence.

**Why this priority**: The handoff is the value bridge to the shipped decision layer, but it depends on Stories 1–3 producing the inputs the interview requires. It is deliberately a **thin boundary**: this feature *hands off*; the interview (spec 121) *runs*.

**Independent Test**: With a committed Stage-1 per-table (Layer-B) profile of the in-scope tables, a proposed scope, and a seeded Decision Store, invoke the handoff. Verify the exact `required_inputs` declared in `contracts/interview/business-knowledge-interview.yaml` are satisfied ("a committed discovery profile" — the per-table Stage-1 profile, not the Layer-A survey; "a proposed scope"; "the existing Decision Store, loaded first"); verify existing decisions are presented for confirmation/supersession and none is overwritten; verify the feature does not itself record interview outcomes, define metric contracts, or grant approvals (those belong to the interview and its downstream).

**Acceptance Scenarios**:

1. **Given** a committed profile, a proposed scope, and an existing Decision Store, **When** the handoff runs, **Then** the interview receives exactly the inputs its contract declares, with the store loaded first, and control passes to the interview (this feature does not re-implement it).
2. **Given** existing Decision Store records for the source, **When** the handoff prepares the interview, **Then** those records are presented for confirmation or supersession and none is silently overwritten.
3. **Given** a suspected-PII column that cannot be sampled safely, **When** the handoff prepares evidence, **Then** the interview's PII stop rule is respected (masked sample or an explicit owner unmasking decision) and no raw value crosses the boundary.
4. **Given** the interview would need a KPI *meaning*, **When** the handoff scopes questions, **Then** meaning questions are marked to route to the Retail KPI knowledge boundary; this feature does not answer them.

### User Story 5 - Route the Next Allowed Action Within the Bounded Discovery Flow (Priority: P3)

Within its own bounded flow — `portfolio discovery -> domain proposal -> scope proposal -> selected-table onboarding -> Business Knowledge Interview handoff -> stop` — the agent consults the existing Knowledge Contracts and committed state to determine the one next allowed action, and performs only that action, then re-evaluates. It uses existing contracts and existing committed state; it does not repair or extend the general cross-stage Decision Gate. Within this bounded flow: no profile blocks the domain proposal; no domain proposal blocks the scope proposal; no proposed scope blocks the interview handoff; unavailable source evidence produces a truthful *local* stop that names the local missing artifact or required user decision; and the flow stops at the Business Knowledge Interview handoff boundary. (The broader guarantee that a specific completely-absent *critical* decision is detected inside a non-empty store is NOT part of this feature — see "Future follow-up: Machine-Readable Required Decisions and Stage Inputs".)

**Why this priority**: Truthful local next-action/stop behavior within the bounded flow is what makes discovery governed rather than a straight-line script. It projects over what Stories 1–4 produce, so it is specified last, but it applies throughout the bounded flow only.

**Independent Test**: Seed committed states along the bounded flow (no profile; profile only; profile + proposed domain; profile + domain + proposed scope) and verify that for each, exactly one next allowed action within the bounded flow is derived from the contracts and committed evidence; that a missing local input (e.g. no profile, no domain proposal) yields a truthful local stop naming what unblocks it; that no state infers a pass from absence; and that a request to cross the interview-handoff boundary stops truthfully.

**Acceptance Scenarios**:

1. **Given** any committed state within the bounded flow, **When** the next action is computed, **Then** the answer is exactly one allowed action within the bounded flow, derived from the relevant stage contract and committed evidence — never two, never "proceed generally".
2. **Given** a missing local input in the bounded flow (no committed profile, or no recorded domain proposal for the scope stage), **When** the dependent stage is evaluated, **Then** the stage's own `required_inputs`/`stop_rules` halt it with a truthful local stop that names the missing local artifact/decision and what would unblock it; absence is never interpreted as a pass.
3. **Given** a local stop, **When** it is reported, **Then** it names the concrete local missing artifact, missing local decision, or unavailable source evidence, and the action that would unblock it.
4. **Given** a pass within the bounded flow, **When** it is reported, **Then** it cites the committed evidence it rests on (a pass with no citable evidence is a defect).
5. **Given** the requested next action would cross the interview-handoff boundary into a downstream capability (KPI contract authoring, Silver/Gold execution, DAX/PBIP generation, dashboard work, publishing), **When** it is evaluated, **Then** the flow stops at the declared handoff boundary and states that the crossing is out of scope for this feature.

### Edge Cases

- **Hundreds of tables**: the Layer-A survey inventories all reachable tables from metadata; any coverage limit is stated with what was omitted (no silent truncation).
- **No obvious fact table**: the survey records the absence as a structural-role observation (from metadata/hints); the domain guess may be low-confidence or undetermined; scope surfaces it as an unresolved dependency — never a fabricated fact table.
- **Multiple tables appear to represent the same business process**: recorded (from name/role hints) as candidate duplicates/overlaps with the competing evidence; not resolved silently. (Value-backed confirmation is a Layer-B check once the tables are in scope.)
- **Candidate relationships from declared FK metadata conflict or are unprovable**: recorded as *candidate from declared metadata* with the conflict noted; never asserted as a proven relationship. Value-backed relationship/orphan proof is a Layer-B check (in-scope tables only).
- **Candidate PK from declared metadata**: recorded as a candidate from declared PK/FK metadata; the survey does NOT test uniqueness/nullability on the rows (that is Layer B). Grain ambiguity is a human seam, not an agent decision (Principle V).
- **Table with no metadata row count (restricted access)**: the metadata item is marked unavailable with the exact reason; the table still appears in the inventory.
- **Metadata access lost after a partial artifact exists**: the partial static evidence is preserved; the missing portions are marked `[PENDING LIVE PROFILE]`/`blocked` with the boundary named; no false pass.
- **File with unknown encoding/delimiter/header/worksheet**: those are `[PROPOSED]` inferences requiring owner confirmation (consistent with Source Ready RS1). At Layer A they are recorded as unconfirmed inferences; any text-column evidence that rests on them arises during Layer-B per-table profiling and is flagged accordingly there.
- **Column with a Layer-A PII suspicion hint**: recorded as a name/type-based hint only; no value is sampled at Layer A. If a masked sample is later required during Layer-B profiling and cannot be produced safely, that per-table check records `needs_sample`; no raw value; dependent verdicts block truthfully.
- **User refuses to expose samples**: the refusal is honored; affected evidence stays masked/`needs_sample`; nothing is fabricated.
- **Existing Decision Store records conflict with newly discovered evidence**: surfaced as a conflict/stale-evidence condition (per spec 121's staleness rule); not silently reconciled. A changed domain/scope proposal is recorded through supersession (a new record `supersedes` the prior; the prior becomes `superseded`), never an in-place mutation.
- **A named human rejects a domain/scope proposal**: recorded as `status: rejected` (the existing terminal refused outcome); the record is preserved, not deleted, and the agent proposes anew only from evidence.
- **A named human accepts only part of a proposed scope**: a NEW bounded `proposed` record covering the accepted subset is created that `supersedes` the original; the original becomes `superseded` so no two active records of the same type share a scope key (FR-019B); the original is never silently mutated.
- **An approved decision cites evidence that has changed or disappeared**: handled by spec 121's existing stale-evidence behavior (blocked for critical types until re-confirmed); this feature does not weaken it. (Domain/scope confirmations are non-critical, so once batch-confirmed they follow the non-critical staleness path — listed under `warn` — not the critical block.)
- **A critical decision is completely missing inside a non-empty store**: detecting this specific case is a **deferred** concern (see "Future follow-up: Machine-Readable Required Decisions and Stage Inputs"), not part of this feature. What this feature relies on is unchanged: PR #257 already fails closed for an absent/empty/malformed store, invalid approvals, conflicts, and missing evidence. Within the bounded flow, a missing local input produces a truthful local stop (FR-024).
- **User stops discovery or interview partway**: whatever valid evidence exists is preserved; unanswered items stay `pending`/`needs_user_input`; nothing is auto-approved.
- **Feature rerun after new tables/columns appear**: new structure is reflected; existing finalized records (approved/rejected/superseded or batch-confirmed) are presented for confirmation, rejection, or supersession and are not overwritten in place; drift is surfaced.
- **Domain classification ambiguous**: recorded as ambiguous/undetermined, never a confident single guess.
- **Scope evidence crosses multiple business processes / fact grains / owner groups / model boundaries**: recorded categorically as **cross-boundary**; narrower coherent options are presented or `needs_user_input` is recorded — no numeric score, table-count threshold, or silent default (User Story 3, AC-2; FR-018).
- **Scale-out (big-data) route requested without recorded scale evidence**: refused; consistent with the flow's existing invariant (bigdata requires recorded scale evidence, else the route is python).
- **Requested next action crosses into a future out-of-scope capability**: the flow stops at the handoff boundary and says so (User Story 5, AC-5).

## Requirements *(mandatory)*

### Functional Requirements

#### Governance invariants (reused, not re-created)

- **FR-001**: The feature MUST reuse the existing seven-stage readiness spine and the existing eleven-stage `contracts/knowledge/database-to-pbip-flow.yaml`. It MUST NOT create a second state machine, approval model, decision vocabulary, or readiness score.
- **FR-002**: The feature MUST NOT grant, infer, or fabricate any human approval. Domain and scope outputs are proposals; critical decisions require explicit named-human approval from an authority class eligible for that decision type per `contracts/knowledge/approval-authority.yaml`. An agent identity never satisfies `approved_by` (Principle V).
- **FR-003**: Agent confidence MUST remain separate from approval and MUST NOT be presented as readiness. The feature MUST NOT emit a numeric readiness or confidence score.
- **FR-004**: The existing `status`, `next`, blocker, approval, and evidence surfaces MUST remain projections of committed truth; the feature's current-stage/next-action output is a projection over them and the contracts, not a new engine.
- **FR-005**: Orchestration MUST remain agent-driven and skill-driven. The feature MUST NOT introduce a daemon, scheduler, background runtime, autonomous approval loop, or a broad new CLI command surface. It is delivered as **a new dedicated skill** (consistent with the ratified Option-B direction and mirroring how spec 121 shipped the `business-knowledge-interview` skill); it adds no new CLI verb (Resolved 2026-07-12).
- **FR-006**: The agent MUST perform only the one next allowed action and then re-evaluate committed state.
- **FR-007**: Every blocked result MUST name the concrete missing artifact, missing decision, invalid approval, or conflicting evidence, and the action that would unblock it. Every pass MUST cite committed evidence.

#### Read-only discovery

- **FR-008**: Discovery MUST be read-only. No DDL, DML, schema mutation, ingestion, transformation, warehouse execution, Power BI execution, or publishing is permitted.
- **FR-009 (Layer A — portfolio survey)**: The multi-table portfolio survey MUST use only read-only source *metadata* and bounded, non-value observations, and MUST cover, per reachable table: source/table identity (without credentials), table/column inventory, declared data types, declared PK/FK metadata where the source exposes it, **candidate grain inferred from declared PK/metadata** (the `discovery` contract's declared "candidate grains" output — a metadata-level candidate, not a value-backed grain proof), approximate/metadata row count or size where safely available, column-name-and-type-based date hints, column-name-and-type-based PII suspicion hints, possible structural role hints (transaction / snapshot / dimension / reference / technical), stated coverage limits and unavailable metadata, candidate domain evidence, candidate tables for the first scope, and for each unavailable metadata item the exact reason it is unavailable. Value-backed grain *proof* (measured on rows) is a Layer-B check (FR-009B). The survey MUST NOT perform deep value-backed measurements (actual missingness, blank-vs-null, measured PK uniqueness/nullability, value-backed grain evidence, value-backed relationship/orphan observations, measured date coverage, masked value samples, returns-column population) for every reachable table; those are Layer-B checks (FR-009B). All Layer-A entries are observations and hints, never approved semantics.
- **FR-009B (Layer B — deep per-table profile, in-scope only)**: Deep, value-backed checks (actual missingness, blank-vs-null observations, measured PK uniqueness and nullability, value-backed grain evidence, value-backed relationship/orphan observations, measured date coverage, masked value samples, returns-column population, and similar per-table mechanical evidence) MUST run ONLY for tables selected into the proposed first scope, and MUST run through the existing per-table onboarding/profiler (`retail-onboard-table` / Source Ready). The feature MUST compose or invoke that existing profiler for in-scope tables; it MUST NOT independently reimplement or duplicate `mappings/<table>/source-profile.md`, and the portfolio survey MUST NOT stand in for it. It is therefore structurally impossible for the future plan to build a second per-table profiler.
- **FR-010**: Portfolio-survey and per-table observations are **evidence, not approved business meaning**. Neither layer MUST assert grain, PK, relationships, PII rulings, or domain as fact; each is recorded as a candidate/observation/hint for human confirmation.
- **FR-011**: Suspected PII MUST be masked by default. Raw suspected-PII values, credentials, secrets, DSNs, and connection strings MUST NEVER be committed to any artifact (Principle IX).
- **FR-012**: When database access, an optional dependency, a source reader, or a measurable property is unavailable, the feature MUST report the unavailable boundary, name what would unblock it, preserve any valid static evidence, use `warning`/`blocked`/`pending`/`needs_sample` semantics as appropriate, never fabricate profile values, and never produce a false pass.
- **FR-013**: The feature MUST NOT re-implement the per-table Source Ready profiler. The two layers are distinct and MUST stay distinct (Resolved 2026-07-12): **Layer A** is a portfolio-level *metadata* survey artifact that precedes per-table onboarding and orients the user across the whole source (FR-009); **Layer B** is the deep value-backed per-table profile, which MUST reuse the existing read-only profiling mechanics by handing each in-scope table off to the existing single-table onboarding (`retail-onboard-table` / Source Ready) rather than duplicating `mappings/<table>/source-profile.md` authoring (FR-009B). The portfolio survey MUST NOT perform Layer-B value-backed profiling, MUST NOT restate or supersede per-table `source-profile.md` truth once a table is onboarded, and MUST NOT be a second per-table profiler.
- **FR-014**: If a coverage limit is applied when discovering a very large source, the limit and the omitted scope MUST be stated explicitly; the feature MUST NOT present a truncated inventory as complete.

#### Domain and scope proposals

- **FR-015**: The domain guess MUST be recorded by the agent as a `proposed` decision (carrying a `confidence` value, per the existing schema) in the existing Decision Store, citing the specific survey facts it rests on. It stays agent-proposed until a named human acts: confirmation is a named-human `batches[]` entry on the low-risk batch path (non-critical, so no authority-class eligibility check and no new authority-map row; FR-019), a refusal makes it `rejected`, and a change supersedes it. The agent MUST NOT confirm it itself.
- **FR-016**: When the domain is ambiguous, the feature MUST record the ambiguity explicitly (competing alternatives with evidence, or an honest "undetermined") and MUST NOT assert a single domain.
- **FR-017**: The first-scope proposal MUST record candidate tables, candidate business questions, candidate KPI *names*, explicitly excluded tables/domains, unresolved dependencies, and the decisions required from the owner — as an agent-authored `proposed` decision (carrying `confidence`) citing survey + domain evidence. It stays agent-proposed until a named human acts: confirmation is a named-human `batches[]` entry on the low-risk batch path (non-critical, so no authority-class eligibility check and no new authority-map row; FR-019), a refusal makes it `rejected`, a change supersedes it, and partial acceptance yields a bounded superseding proposal (FR-019B). The agent MUST NOT confirm it itself.
- **FR-018**: Scope bounding MUST be deterministic and evidence-based. The scope proposal MUST honor an explicit user-supplied scope limit when one exists; otherwise it MUST prefer one coherent business process or subject area, one primary fact grain, and candidate KPIs that can share a coherent model boundary. It MUST identify when the proposal crosses multiple business processes, fact grains, unrelated owner groups, or incompatible model boundaries, and MUST then present narrower candidate options where the evidence supports more than one coherent slice, or record `needs_user_input` when a human choice is required. A scope MAY be described categorically only as **coherent**, **cross-boundary**, **unresolved**, or **needs user input** (these are descriptive prose, not a stored scale). The feature MUST NOT use a numeric scope score, a hidden table-count threshold, a fabricated priority rank, or a silent default; MUST NOT claim a scope is objectively correct from structural evidence alone; MUST NOT silently enlarge scope; and MUST NOT invent another readiness or confidence scale.
- **FR-019**: Domain and scope proposals MUST enter the **existing** Decision Store using the existing decision vocabulary and lifecycle; the feature MUST NOT introduce a parallel store or a second vocabulary. They are recorded as **non-critical proposals confirmed within the interview** (Resolved 2026-07-12): NO new critical `decision_type` is added and NO new row is added to `approval-authority.yaml` — 121's critical-decision vocabulary and authority map stay frozen. This is structurally supported by the existing Decision Store record schema, which already admits **non-critical free-form `decision_type` tokens via its pattern branch** (only critical types are enumerated; only non-critical types may be batch-approved) — so no schema change is required. Two structural constraints keep "non-critical / no new authority row" true and MUST hold:
  - the domain/scope record's `decision_type` MUST NOT appear in any stage's `blocking_decision_categories` (the `discovery`/`domain_guess`/`scope_proposal` stages already declare `blocking_decision_categories: []`), so 121's fail-closed rule for "an unknown type **inside a blocking category**" is never triggered by these records; and
  - confirmation MUST use the interview's low-risk **batch** confirmation path — a `batches[]` entry recording `confirmed_by` (a named human plus an authority class, RS1 shape), `members`, and the presented `evidence`, with each member carrying `batch_id`. Because the type is non-critical, this path triggers NO authority-class eligibility check against `approval-authority.yaml` and needs NO new authority-map row (121's DS2 eligibility check applies to critical types only). It MUST NOT route a domain/scope proposal through the critical per-decision approval path (the reason a naive `status: approved` flip is wrong: DS2 fires the full approval-completeness requirement for ANY `approved` record regardless of criticality — so confirmation must go through the low-risk batch path, not a bare status change), MUST NOT add the type to `approval-authority.yaml`, and MUST NOT self-grant (the agent is never `confirmed_by`/`approved_by`).
  The domain/scope lifecycle uses ONLY existing statuses and record fields (there is NO `confirmed` status and NO second lifecycle). Spec 122 introduces no new status and no new confirmation mechanic; the exact recorded status of a batch-confirmed member follows 121's existing convention:
    1. **Agent proposes** — a `proposed` record with `confidence`, cited evidence, `proposed_by` = the agent identity, and a `scope` selector; it is not an approval.
    2. **Named human accepts** — the record becomes a `members` entry of a `batches[]` confirmation whose `confirmed_by` is the named human + authority class (never the agent). The member's resting status follows 121's existing batch convention; either way its `decision_type` sits in no `blocking_decision_categories`, so it gates no downstream stage.
    3. **Named human rejects** — the record's `status` becomes `rejected` (the existing terminal "refused" outcome); it is never deleted or overwritten.
    4. **Named human accepts only part of a scope** — see FR-019B (a bounded superseding proposal, never a silent mutation of the original).
    5. **Proposal changed after new evidence** — the changed proposal is a NEW record that `supersedes` the prior one, and the prior record's `status` becomes `superseded` with `superseded_by` set (the existing supersession behavior); history is preserved.
    6. **Existing finalized records rerun** — presented for confirmation, rejection, or supersession and NEVER overwritten in place.
  A domain/scope proposal is therefore never auto-approved by the agent and never self-granted. The critical decisions that DO gate downstream stages (grain, PII, KPI meaning, etc.) remain 121's existing types, unchanged, and still require the full critical per-decision approval (including authority-class eligibility).
- **FR-019B**: Partial acceptance of a scope MUST produce a clearly bounded resulting proposal (a NEW `proposed` record whose `scope` covers only the accepted subset and which `supersedes` the original), never a silent mutation of the original record. When the narrower record becomes the active proposal, the original MUST be marked `superseded` (with `superseded_by` set) so that no two active records of the same type share a scope key (the existing store forbids conflicting active records on one scope). Rerunning the feature over existing finalized (approved/rejected/superseded or batch-confirmed) records MUST present them for confirmation, rejection, or supersession and MUST NOT overwrite them in place.

#### Handoff to the Business Knowledge Interview

- **FR-020**: The feature MUST hand the committed discovery profile and the proposed scope into the already-shipped Business Knowledge Interview, satisfying exactly the `required_inputs` declared in `contracts/interview/business-knowledge-interview.yaml` ("a committed discovery profile", "a proposed scope", and "the existing Decision Store, loaded first"). The interview's "committed discovery profile" input MUST be satisfied by the Stage-1 per-table (Layer-B) profile of the in-scope tables (the interview contract names it "Stage-1 read-only profile of `retail-onboard-table` or equivalent"); the Layer-A portfolio survey MUST NOT be substituted for it. It MUST NOT invent inputs the interview does not accept, and MUST NOT re-implement the interview.
- **FR-021**: Interview questions handed off MUST be grounded in discovery evidence. Existing Decision Store decisions MUST be loaded first and presented for confirmation or supersession, never overwritten.
- **FR-022**: The feature MUST NOT itself record interview outcomes, define metric contracts, define KPI meaning, or grant approvals; those are owned by the interview and its downstream. KPI-meaning questions MUST route to the existing Retail KPI knowledge boundary.
- **FR-023**: The feature MUST clearly define and observe the handoff boundary to the later `kpi_contracts` stage: it stops before KPI contract authoring. The interview stage's declared handoff (`kpi_contracts`) is downstream and out of scope for this feature.

#### Truthful local-stop behavior within the bounded discovery flow

The requirements below are scoped to this feature's own bounded flow (`portfolio discovery -> domain proposal -> scope proposal -> selected-table onboarding -> interview handoff -> stop`). This feature does NOT repair, extend, or replace the general cross-stage Decision Gate; the broader machine-readable-required-inputs and missing-specific-critical-decision work is deferred (see "Future follow-up: Machine-Readable Required Decisions and Stage Inputs").

- **FR-024**: Within the bounded flow, unavailable source evidence or a missing local input MUST produce a truthful *local* stop that names the concrete local missing artifact or required user decision and what would unblock it. Absence MUST NEVER be interpreted as approval, pass, not-applicable, or low-risk. Sequencing within the flow uses each stage's existing `required_inputs`/`stop_rules` (no profile -> domain proposal blocked; no domain proposal -> scope proposal blocked; the scope proposal does not block the interview handoff).
- **FR-025**: This feature MUST NOT weaken the existing fail-closed verdict behavior it inherits from spec 121 / PR #257: a missing, malformed, unreadable, empty, conflicting, stale, or invalid store or approval already blocks (never passes). This feature relies on that behavior unchanged; it neither re-implements nor extends it.
- **FR-026**: The feature MUST rely on the already-declared `discovery` / `domain_guess` / `scope_proposal` entries in `contracts/knowledge/database-to-pbip-flow.yaml` (which already declare their `required_inputs`, `stop_rules`, and `blocking_decision_categories`) **without modifying the contract schema or those entries**, and MUST NOT add a parallel evaluation mechanism or a second readiness engine. It MUST NOT make all eleven stages' `required_inputs` machine-readable, MUST NOT modify the general Decision Gate, and MUST NOT change all-stage readiness behavior — those are deferred (future follow-up).
- **FR-027**: The current-stage/next-action projection MUST derive the one next allowed action **within the bounded flow** from the existing contracts and committed state, introducing no second readiness engine or parallel workflow state. It MUST NOT compute a global next action across all eleven stages and MUST NOT guarantee cross-stage absence detection outside this bounded flow.

#### Explicit non-production boundaries

- **FR-028**: The feature MUST NOT write Silver or Gold SQL, MUST NOT define KPI meaning, and MUST NOT generate DAX, metric contracts, dashboard visuals, PBIP artifacts, or Power BI reports.
- **FR-029**: The feature MUST preserve approved decisions as immutable records; any change MUST use supersession (reusing spec 121's supersession behavior, not a new mechanism).
- **FR-030**: A scale-out (big-data) route MUST NOT be selected without recorded evidence that the source exceeds single-node capacity; absent that evidence the route is the single-node (python) path, consistent with the flow's existing boundary invariant.

## Key Entities

- **Multi-Table Portfolio Survey (Layer A)**: One committed, reviewable artifact surveying all reachable tables of a source from read-only *metadata* — the `required_outputs` of the `discovery` contract, at portfolio scale. Contains metadata facts (identity, inventory, declared types, declared PK/FK metadata, approximate/metadata row count), name-and-type-based date and PII *hints*, structural role hints, candidate domain evidence, candidate first-scope tables, and explicitly-marked unavailable metadata with reasons. It holds NO value-backed measurement (no measured uniqueness/missingness/date-coverage, no raw or masked samples) — those belong to Layer B. Evidence and hints, never approved meaning.
- **Deep Per-Table Profile (Layer B)**: For each table selected into the proposed first scope, the value-backed profile produced by the existing per-table onboarding/profiler (`retail-onboard-table` / Source Ready) — measured missingness, blank-vs-null, PK uniqueness/nullability, value-backed grain and relationship/orphan evidence, measured date coverage, masked samples, returns-column population. This feature invokes that profiler; it is NOT a new artifact this feature authors and it never duplicates `mappings/<table>/source-profile.md`.
- **Domain Proposal**: An agent-authored `proposed` decision in the existing Decision Store (non-critical free-form `decision_type`) naming a candidate retail subject area, citing survey evidence, carrying the agent's `confidence` (`low`/`medium`/`high` — separate from approval, never a readiness signal). A named human confirms it via the low-risk **batch** path (a `batches[]` entry; non-critical, so no authority-class eligibility check and no new authority-map row — the confirmed member's recorded status follows 121's existing convention), rejects it (`status: rejected`), or supersedes it. The agent never self-confirms (FR-019).
- **First-Scope Proposal**: An agent-authored `proposed` decision in the existing Decision Store (non-critical free-form `decision_type`) bounding a first scope — candidate tables, candidate questions, candidate KPI names, exclusions, unresolved dependencies, required owner decisions — citing survey + domain evidence, carrying `confidence`, and following the same lifecycle (named-human `batches[]` confirmation on the non-critical batch path; reject; supersede; partial acceptance produces a bounded superseding proposal, FR-019B).
- **Handoff Package**: The set of inputs this feature presents to the Business Knowledge Interview, matching the interview contract's declared `required_inputs` exactly (committed profile, proposed scope, existing Decision Store loaded first). Not a new artifact type where the contract already names the inputs; it is the act of satisfying them.
- **Current-Stage / Next-Action Projection (bounded flow)**: A read-only view derived from the existing contracts and committed state (over the shipped `status`/`next`/`blockers` surfaces) reporting, **within this feature's bounded discovery-to-handoff flow**, the current stage, the one next allowed action, local blockers, required approval, and the required stop point. Holds no independent state and does not compute a global all-stage next action or repair the general Decision Gate.
- **Blocker Explanation**: The concrete naming of a missing artifact / decision / invalid approval / conflicting evidence and its unblocking action (reuses the existing blocker surface).
- **Evidence & Approval References**: Pointers from proposals/verdicts to the committed profile facts and (for approved decisions) the approval metadata — reusing spec 121's approval-record shape; no new metadata model.

> **Artifact-shape decision (separate artifacts vs one package)**: Per the contract's own `required_outputs`, the Layer-A portfolio survey is **one new artifact** (multi-table, metadata only); the Layer-B deep per-table profiles are **not new artifacts this feature authors** — they are produced by the existing per-table profiler for in-scope tables (`mappings/<table>/source-profile.md`); the domain and scope proposals are **records in the existing Decision Store** (not new files); the current-stage/next-action output is a **projection view** (not a stored artifact); blocker/evidence/approval references reuse existing surfaces. This avoids duplicate truth: the portfolio survey is the single home for portfolio-level metadata observations, the existing per-table profiler is the single home for value-backed per-table truth, the store is the single home for decisions, and the projection derives everything else. The precise packaging (one survey file vs a small survey-plus-index set) is a plan-time layout choice that does not change this ownership split.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A new external user can provide a real multi-table retail source (>= 5 tables) and receive one reviewable Layer-A portfolio survey without reading the whole repository — verified by a moderated walkthrough where the user locates every required survey element (metadata facts, hints, coverage limits, candidate domain/scope evidence) from the single artifact.
- **SC-002**: For a worked-example source, the Layer-A portfolio survey plus the Layer-B profiles of the in-scope tables are together sufficient to conduct a focused Business Knowledge Interview with **fewer question rounds than the in-scope column count** (the survey orients across the source and the in-scope per-table profiles pre-answer the mechanical questions, so the interview concentrates on meaning). The survey alone does not pre-answer value-backed mechanical questions — those come from the Layer-B profiles of the selected tables.
- **SC-003**: Agent-authored domain and scope proposals are clearly distinguished from human confirmation — 100% of agent-authored records read `status: proposed` with `proposed_by` = the agent and are never self-confirmed by the agent; any acceptance is carried by a named-human `batches[]` confirmation (`confirmed_by` = a named human + authority class) on the non-critical batch path that adds no `approval-authority.yaml` row and applies no authority-class eligibility check. No agent identity ever appears as `confirmed_by`/`approved_by`.
- **SC-004**: Zero raw suspected-PII values, credentials, secrets, DSNs, or connection strings appear in any committed artifact across all fixtures.
- **SC-005**: Within the bounded flow, a missing local input (no committed profile; no recorded domain proposal for the scope stage) produces a **deterministic local stop** naming the missing local artifact/decision — 100% of seeded bounded-flow "missing local input" cases stop (no false pass, no absence-as-approval). The inherited PR #257 fail-closed behavior for an absent/empty/malformed store, invalid approval, or conflict is relied on unchanged. (Detecting a specific completely-absent *critical* decision inside a non-empty store is deferred — see the Future follow-up section — and is NOT asserted as a success criterion of this feature.)
- **SC-006**: Existing finalized decisions (approved, rejected, superseded, or batch-confirmed) are preserved and any change occurs only through supersession — 0 in-place edits or overwrites across rerun fixtures; partial scope acceptance always yields a bounded superseding record, never a mutation of the original.
- **SC-007**: From any seeded committed state **within the bounded discovery flow**, the agent identifies **exactly one** next allowed action from committed evidence and the contracts (never zero-when-work-remains-in-the-flow, never more than one). This is scoped to the bounded flow; it does not assert a global all-stage next-action computation.
- **SC-008**: The workflow stops before KPI contract authoring, warehouse execution, dashboard creation, PBIP generation, and publishing — verified by inspection of delivered artifacts (none produced).
- **SC-009**: The feature duplicates no capability already shipped on `main` — verified against `docs/capabilities/capabilities.yaml` (no discovery/domain/scope producer exists today) and against spec 121 (which declares these stages but scopes their automation out).
- **SC-010**: Every acceptance criterion is observable and testable without relying on subjective confidence or a numeric readiness score.

## MVP Boundary

- **MVP = User Story 1 (P1)**: the read-only Layer-A multi-table portfolio survey (metadata + hints across all reachable tables). It is the input every later story depends on, it is independently testable, and it delivers standalone value (a legible orientation across an unfamiliar source). Deep value-backed per-table profiling (Layer B) is confined to in-scope tables and runs only in the downstream scope step (User Story 3, P2) through the existing per-table profiler — so **no deep profiling occurs at MVP**.
- **Later enhancements**: User Story 2 (domain guess) and User Story 3 (scope proposal) at P2; User Story 4 (interview handoff) and User Story 5 (bounded-flow next-action routing) at P3. These are separable slices layered on the MVP survey.

## Future follow-up: Machine-Readable Required Decisions and Stage Inputs (NOT this feature)

This is a real follow-up surfaced by PR #257, deliberately **out of scope** for spec 122 and **non-blocking** for it. Spec 122 is a discovery-to-interview-handoff feature; it must not expand into a global repair of the Decision Gate. The following belong to a separate future specification and MUST NOT be built in this task or this feature:

- making all eleven stages' `required_inputs` machine-readable;
- deterministically detecting a specific **completely-absent critical decision inside a non-empty store**;
- cross-stage Decision Gate behavior and global next-action computation across all eleven stages;
- guaranteeing cross-stage absence detection outside spec 122's bounded flow.

What is already true on `main` and is relied on unchanged: PR #257 already **fails closed** for an absent or empty Decision Store, malformed/unreadable records, invalid approvals, conflicts, and missing/unresolvable evidence. The gap that a separate future spec may address is narrow — deterministically detecting that one *specific* critical decision is missing from an otherwise valid, non-empty store. That future spec is not created here.
