# Phase 0 Research: Live-Surface Import Boundary Guard (B3)

All findings are grounded against the committed repo on branch
`048-live-surface-import-boundary-guard`. No open technical unknowns remain;
remaining open items are the governance/scope decisions reserved for human
ratification in spec.md ## Clarifications.

## R1 -- Reusable surface of `never_execute.py`

**Decision**: Reuse, unchanged, from `src/retail/rules/never_execute.py`:
`module_scope_violations(source: str) -> list[str]`, plus `_FORBIDDEN_ROOTS`,
`_FORBIDDEN_DOTTED`, `_is_forbidden`, and the SyntaxError-to-Finding pattern B1
uses in its rule body.

**Rationale**: `module_scope_violations` already does exactly what B3 needs:
stdlib `ast.parse` over source text; visits the top-level body plus module-level
`try`/`if` bodies; ignores `def`/`async def`/`class` bodies (lazy imports) and
`if TYPE_CHECKING:` blocks; returns the forbidden module-scope import names. It
raises `SyntaxError` only when the source does not parse, which the caller turns
into a Finding. The forbidden sets (psycopg2, psycopg, asyncpg, sqlalchemy,
pymysql, pyodbc, requests, httpx, aiohttp, socket, http, urllib3, ftplib,
smtplib, telnetlib, websocket, websockets; plus `urllib.request`/`urllib.error`
but NOT `urllib.parse`) are precisely the connection-capable set B3 wants. Reuse
honors Principle II (Depend, Never Fork) and the stdlib-only invariant.

**Alternatives considered**: (a) Copy the helper into a new module -- REJECTED
(forks the parser and forbidden list; two sources of truth drift). (b) Generalize
B1 to take the module set as a parameter and call it twice -- viable but larger
than the first-step scope; deferred. The chosen path imports the existing helper
and adds only a thin rule wrapper + a new module-set constant.

## R2 -- Where the new rule lives

**Decision**: Add a new sibling rule module
`src/retail/rules/live_surface_boundary.py` that imports the shared helper and
forbidden sets from `never_execute` and registers the new rule. (Equivalent
fallback if the team prefers co-location: a second `@register` function inside
`never_execute.py`. tasks.md uses the sibling-module path as primary; the
contract is identical either way because both reuse the same helper.)

**Rationale**: A small, single-purpose module keeps cohesion high and matches the
repo's "many small files" coding-style rule; it also keeps B1's module
(static-core policy) textually distinct from B3's module (live-surface policy),
which are deliberately inverse policies over disjoint module sets. `pkgutil`
auto-discovers any new submodule under `retail.rules`, so the wiring test's
"all submodules importable" check picks it up automatically.

**Alternatives considered**: Co-locating inside `never_execute.py` -- acceptable
and slightly fewer files, but mixes two inverse policies in one module. Either
satisfies the contract; the sibling module is preferred for clarity.

## R3 -- Registration + wiring contract

**Decision**: Register the rule with `@register(<id>, <title>)` (the B-family id
is reserved for human ratification -- see spec ## Clarifications). Add that id to
`EXPECTED_RULE_IDS` in `tests/unit/test_rules_wiring.py` and regenerate
`docs/rules/rules-manifest.json` via `retail manifest` in the same change.

**Rationale**: `tests/unit/test_rules_wiring.py` is the single source of truth for
the registered id set. Its snapshot test clears `registry._RULES`, reloads every
submodule via `pkgutil`, and asserts the live id set equals `EXPECTED_RULE_IDS`
AND that `len(all_rules()) == len(EXPECTED_RULE_IDS)`. The count is derived from
the set's own length -- never a literal number -- so B3 adds exactly one id and
hard-codes no baseline. The 043 rule-registry snapshot test guards
`docs/rules/rules-manifest.json`, regenerated by the `retail manifest` CLI
subcommand (`cli.py`'s `_run_manifest`). Both must be updated together or the
snapshot test fails.

**Alternatives considered**: Hard-coding a baseline count (e.g. "now 34") --
REJECTED; the committed set already changed (A1, A3, B1, G6, ... were added over
time) and any literal drifts. The spec/plan therefore never assert a baseline.

## R4 -- The wiring-latent-gap obligation

**Decision**: Beyond listing the id, add a direct unit test that invokes the rule
against a synthetic known-bad live-surface source and asserts a non-empty Finding
set (the rule fires), plus a known-good (lazy import) source that asserts no
Finding.

**Rationale**: Repo memory records a prior gap where a registered rule was listed
in the wiring set but never actually exercised, so a no-op rule could pass
silently. FR-009 / SC-004 require the rule to be exercised firing. The direct
test closes that gap independently of the snapshot test.

**Alternatives considered**: Relying solely on the snapshot test -- REJECTED; it
proves the id is registered, not that the rule produces a Finding on a violation.

## R5 -- The live-surface module set

**Decision**: Define an explicit constant of repo-relative POSIX paths --
candidate membership `{src/retail/validate.py, src/retail/value_proxy.py,
src/retail/semantic.py, src/retail/dax_gen.py}` -- disjoint from B1's
`_GOVERNED_MODULES` (`cli`, `runner`, `core`, `registry`) and `_GOVERNED_PREFIX`
(`src/retail/rules/`). Final membership is reserved for human ratification (spec
## Clarifications).

**Rationale**: Verified disjointness: none of the four live-surface modules is in
B1's governed set or under the `rules/` prefix, so the inverted-policy framing
holds and the two rules never double-cover a module. All four modules today keep
their driver/`yaml` imports lazy (confirmed by reading their module-scope
imports), so the rule is green on the current tree and only fires on a future
regression.

**Alternatives considered**: Deriving the set dynamically (e.g. "any module that
imports `QueryRunner`") -- REJECTED for the first step; it is implicit, harder to
audit, and could silently widen scope. An explicit, ratifiable closed set is the
YAGNI choice; widening it later is a one-line change.
