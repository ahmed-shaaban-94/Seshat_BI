# Phase 0 Research: Scaffold-Rule Authoring Generator + Doctor

## R1 -- The five wiring places (confirmed against the live repo)

A new governance rule must appear in five places to be fully wired. Each was
verified in the current worktree:

1. **Registered rule module** -- a `@register(rule_id, title)` decorator in a
   `src/retail/rules/*.py` module. Mechanism confirmed in `src/retail/registry.py`
   (`register()` appends a `RegisteredRule` to `_RULES`; `all_rules()` returns
   them). Minimal, stdlib-only.
2. **Module import list + public export** -- the new module must be added to BOTH
   the side-effecting `from . import (...)` block AND `__all__` in
   `src/retail/rules/__init__.py`. Confirmed (import block + `__all__` both list
   every rule module).
3. **Expected-id source-of-truth set** -- the new id must be inserted into the
   `EXPECTED_RULE_IDS` frozenset in `tests/unit/test_rules_wiring.py`; the wiring
   test asserts the registered set equals this set (count is keyed to its length).
   Confirmed present.
4. **Two golden records** -- `manifest.py` -> `docs/rules/rules-manifest.json`
   (CLI `manifest`) and `severity_posture.py` -> `docs/rules/severity-posture.json`
   (CLI `severity-posture`). Both are regenerated by a PRINTED command, guarded by
   snapshot tests. Confirmed both subcommands exist in `cli.py`.
5. **Rule-family prose row** -- a row in `docs/glossary.md`. Confirmed the
   glossary carries rule-family rows (e.g., the SL and AL families).

## R2 -- The drift is real and current

The most recently added design-lint rule is registered, present in
`EXPECTED_RULE_IDS`, and present in the generated manifest, yet has NO
rule-family row in `docs/glossary.md` (the glossary's newest family rows do not
include it). This is a live, current instance of place #5 drifting -- exactly the
failure mode Doctor exists to surface. It is cited generically (as "a registered
rule with no glossary row") so the feature carries no dependence on that specific
rule's identity.

## R3 -- Existing generators are stdlib-only and never execute

`manifest.py` and `severity_posture.py` are the two existing "generator"
modules. They read the live registry / force rules and write a golden JSON; they
are pure stdlib and do not stand up a database, a network call, or a Power BI
runtime. The new helper matches this posture exactly (Principle VIII): it reads
and writes repo text only.

## R4 -- The ceremony has no machine-usable manifest today

The five-place list exists only as a private memory note; there is no
machine-readable declaration of "the ceremony" in the repo. This is the gap the
feature closes. Two options were considered:

- **(a) Hardcode the list in the helper.** Simple; but a hardcoded list can itself
  drift from the real repo -- the exact meta-risk the idea bank flagged.
- **(b) Derive the list dynamically from the repo.** No drift, but materially
  larger scope (repo-structure introspection) for a first step.

**Decision**: (a) with a guard (the declared list PLUS a test that fails if the
repo grows a place the list does not name). This keeps the first step thin
(YAGNI) while structurally preventing the hardcoded list from silently drifting.
Dynamic discovery is deferred to a later step if it ever proves needed.

## R5 -- Write vs print boundary (the load-bearing decision)

The generator's outputs split into two disjoint sets:

- **WRITE** (safe, mechanical, no judgment): the stub rule module, the failing
  test stub, and the membership insertion into `EXPECTED_RULE_IDS`.
- **PRINT-only** (carries regeneration or editorial semantics): the two golden
  regen commands and the glossary prose row.

Rationale: inserting an id into a membership set is a mechanical edit an author
is entitled to automate. Regenerating a golden record is a build action that
rewrites a source-of-truth artifact; editing prose applies human editorial
judgment (wording, family, ordering). Both stay human-run/human-pasted so the
helper never mutates a golden record or fabricates intent (Principles V, VIII).

## R6 -- Exit-code contract for Doctor

Doctor is a report, but a report is only useful in CI if its exit code is
deterministic. Decision (stated in the CLI contract): Doctor exits **non-zero
when it finds drift** (any checked id missing from any place) and **zero when all
checked ids are present in all five places**; an unknown/unregistered id is
reported but does not itself crash-exit. This lets a human or CI job gate on
"no wiring drift" without parsing text.

## R7 -- Generic-only stub (Principle VII)

The generated stub rule body must be a minimal, always-safe placeholder (a rule
that yields no findings) with zero worked-example specifics. The generated test
stub asserts something that fails until real logic is written (honest red). No
example table names, columns, codes, or report paths appear anywhere in the
generated output.
