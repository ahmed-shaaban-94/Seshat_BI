# Phase 1 Data Model: Rule-Count Claim Reconciler (SC2)

SC2 introduces no new core types. It reuses the existing rule contract
(`Finding`, `Severity`, `RuleContext`) and adds one new human-curated data file
(the rule-count-claim manifest). This document describes that file's schema, the
authoritative count source it reconciles against, and the existing types SC2
consumes.

## New entity: Rule-count-claim manifest

**File**: `docs/quality/rule-count-claims.yaml` (committed, tracked, hand-curated).

**Top-level shape**: a YAML mapping with a single key `claims` whose value is a
list of claim records. An explicitly empty list (`claims: []`) is valid (honest
"nothing to check yet"); a missing/untracked file, non-YAML content, or a shape
that is not a mapping-with-a-`claims`-list is an ERROR (fail-loud).

### Claim record fields

| Field           | Type                 | Required | Meaning |
|-----------------|----------------------|----------|---------|
| `id`            | string               | yes      | Stable handle for the claim, named in findings. |
| `doc`           | string (POSIX path)  | yes      | Repo-relative path of the document making the count claim. MUST be a tracked file. |
| `anchor`        | string               | yes      | Verbatim snippet expected to be present in `doc`. Presence-checked as a substring so a moved/removed sentence is itself flagged. |
| `claimed-count` | integer (>= 0)       | yes      | The rule count the prose asserts. Reconciled against the authoritative count. |

**Notes**:
- All four fields are required; a record missing any one is an ERROR (SC2 never
  guesses a missing field).
- `claimed-count` is the ONLY source of the integer SC2 compares. SC2 does NOT
  extract a number from `anchor` text.
- The manifest is scoped by authoring discipline to LIVE-STATE claims only; dated
  as-of-then snapshots MUST NOT be listed (see spec FR-017). SC2 enforces this
  structurally by being manifest-only.

## Authoritative count source (read, not authored by this feature)

**File**: `docs/rules/rules-manifest.json` (committed, tracked, regenerated by
`retail manifest`).

**Shape consumed**: a JSON array of rule objects; SC2 derives the authoritative
count as the number of entries (the array length). This file is golden-tested
(`tests/unit/test_rules_manifest_snapshot.py`) to equal `len(all_rules())`, so its
length is an accurate proxy for the live registry count. SC2 reads it with the
standard-library `json` module; a missing/untracked/unparseable source is an ERROR.

## Reused core types (UNCHANGED)

- **`Finding`**: `{rule_id, severity, message, locator}`. SC2 emits ERROR findings
  only, with `rule_id="SC2"`.
- **`Severity`**: SC2 uses `Severity.ERROR` exclusively.
- **`RuleContext`**: read-only context handed to every rule -- provides
  `repo_root` (a `Path`, used to read the manifest, each `doc`, and the count-source
  JSON as committed text) and `tracked_files` (repo-relative POSIX paths, used to
  confirm the manifest, the count source, and each `doc` are tracked). SC2 adds no
  new context field.

## Derived-value discipline (Hard rule 9)

The only integers SC2 handles are the declared `claimed-count` and the
authoritative count (an array length). A finding names both integers to explain the
mismatch. SC2 computes no ratio, percentage, score, or graded value -- the
comparison is categorical equality.
