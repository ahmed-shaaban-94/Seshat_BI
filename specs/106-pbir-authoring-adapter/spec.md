## Context and boundary (read first)

This feature **completes the deliberately-unbuilt second half of F034** (Visual
Implementation MVP). Today the kit **forbids agent-generated PBIR** (FR-008/FR-009
of spec 001; `dashboard-design/SKILL.md:78` "STOP before authoring PBIR via
automation"); the committed `powerbi/*.Report/` page is left empty for a human to
build in Power BI Desktop. This feature proposes a **companion authoring adapter**
-- the F029-dbt / F030-Dagster pattern -- that writes the PBIR JSON itself, under a
tight boundary, **only after the owner ratifies the FR-008/009 lift.**

**Slice 1 (the theme generator, `retail theme-gen`) shipped separately (PR #204).**
This adapter *consumes* the theme it produces; it does not re-implement theming.

**Honesty guard (carried from the design review that shaped this):** this feature
builds the *mechanism* -- programmatically writing visual formatting into PBIR JSON.
It does NOT deliver "great creative professional dashboards" automatically; that is
a separate design-intelligence layer that would ride on top. The spec MUST NOT
conflate the mechanism with the dream.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Apply a generated theme's formatting to report visuals (Priority: P1)

An analyst has a committed PBIR report (visuals present but unstyled or default-
styled) and a theme produced by `retail theme-gen`. They run the adapter; it writes
the theme-derived formatting (card fill/border/title font, chart data colors, label
defaults) into the report's `visual.json` files -- the same settings a human would
click through in the Power BI UI -- deterministically and reviewably (a git diff).

**Why this priority**: This is the core mechanism and the smallest end-to-end slice
that delivers the user's actual ask ("change settings of cards like a human does").
Everything else is an extension of it.

**Independent Test**: Given a fixture PBIR report + a generated theme, run the
adapter; assert the visual.json files now carry the theme's colors/fonts, the diff
is deterministic (byte-identical on re-run), and `retail check` (R1 + the new
authoring-lint) stays green.

**Acceptance Scenarios**:

1. **Given** a committed PBIR report and a generated theme, **When** the adapter
   runs, **Then** each targeted visual's formatting keys are set from the theme and
   the write is idempotent (re-running produces no diff).
2. **Given** a visual whose type is not in the allowed-edit list, **When** the
   adapter runs, **Then** it leaves that visual untouched and records why.
3. **Given** the FR-008/009 lift has NOT been ratified, **When** the adapter is
   invoked, **Then** it refuses to write and prints the gate (Principle V).

---

### User Story 2 - Apply a background as surface-2 output (Priority: P2)

The adapter sets a page's canvas background to a committed surface-2 background
asset (referenced by path from the page blueprint / `background-spec.yaml`), writing
the page-background reference into the PBIR page JSON -- never baking data into an
image.

**Why this priority**: Backgrounds are the user's second explicit ask; they extend
the same PBIR-writing mechanism to the page surface. Lower than P1 because a report
is usable (themed) without a custom background.

**Independent Test**: Given a page + a committed background asset, run the adapter;
assert the page JSON references the asset with the correct fit/transparency and the
surface-2 purity rule holds (no KPI value / dynamic content baked in).

**Acceptance Scenarios**:

1. **Given** a committed background asset and a page, **When** the adapter runs,
   **Then** the page background reference is written with fit=Fit, transparency=0.
2. **Given** a background asset that fails the surface-2 purity checks, **When** the
   adapter runs, **Then** it refuses to reference it and records the reason.

---

### User Story 3 - Validate and round-trip every write (Priority: P1, cross-cutting)

Every PBIR write is validated: the written JSON is schema-checked, the report still
opens (structurally), R1 (model-reference integrity) stays green, and a round-trip
(read -> write -> read) is stable. A failed validation rolls back (writes nothing).

**Why this priority**: A file-writing adapter that can corrupt a report is worse
than no adapter. Validation is co-P1 with US1 -- the mechanism is only safe if every
write is verified.

**Independent Test**: Given an edit that would produce invalid PBIR, run the
adapter; assert it writes nothing, exits non-zero, and names the failure.

**Acceptance Scenarios**:

1. **Given** any adapter write, **When** it completes, **Then** the result is valid
   PBIR JSON and re-reading it yields the same model (round-trip stable).
2. **Given** a write that would break R1 (absolute/remote model path), **When** the
   adapter runs, **Then** it refuses and rolls back.

---

### Edge Cases

- What happens when the target report has no visuals, or a visual is missing the
  expected formatting container? (Adapter records a `warning`, writes nothing for
  that visual, does not fabricate a container.)
- What happens on a partial failure mid-batch? (All-or-nothing per report: stage
  writes, validate, then commit; a failure rolls back the whole batch.)
- What happens when the theme and the report's model disagree (a data color for a
  series that no longer exists)? (Applies theme-level defaults; per-series binding
  that has no target is recorded, not invented.)
- What happens when a visual type is outside the allowed-edit allow-list? (Untouched
  + recorded -- the adapter never edits a property it has not declared safe.)
- What happens if invoked with no ratified FR-008/009 lift? (Hard refuse -- the
  gate.)

## Requirements *(mandatory)*

### Functional Requirements -- the adapter boundary

- **FR-001 (adapter boundary)**: The adapter MUST be a companion authoring adapter
  under `.claude/skills/` (the F029/F030 pattern), NOT part of the static
  DEFINE/CHECK core (`src/retail/` rules). It reads committed artifacts and writes
  committed PBIR/PBIP JSON; it defines no metric, mapping, or semantic logic.
- **FR-002 (allowed PBIR edits -- an allow-list)**: The adapter MUST edit ONLY a
  declared allow-list of visual/page formatting properties: card fill, border,
  title text/font/alignment, data-label defaults, gridline style, data colors, page
  background reference. Any property not on the allow-list MUST NOT be written. The
  allow-list is a committed, guard-tested constant (it cannot silently grow).
- **FR-003 (forbidden edits)**: The adapter MUST NOT edit data bindings, measures,
  DAX, relationships, filters that change meaning, or any semantic-model content --
  formatting only. It MUST NOT create a data-bound visual with no backing approved
  contract (the surface-1 orphan rule).
- **FR-004 (no external dependency)**: The adapter MUST NOT use pbi-cli, the Power
  BI MCP, a live Power BI/workspace connection, or any network call. It MUST be
  stdlib + `pyyaml`-level dependency-light (JSON is stdlib). It operates on local
  committed files only.
- **FR-005 (deterministic + reviewable)**: Every write MUST be deterministic (same
  inputs -> byte-identical output) and produce a reviewable git diff. No timestamps,
  no random ordering, stable key ordering.
- **FR-006 (safety checks)**: The adapter MUST (a) validate every input path stays
  within the repo (no traversal), (b) refuse to overwrite without an explicit
  intent flag, (c) all-or-nothing per report (stage -> validate -> commit or roll
  back), (d) never write a partial/corrupt file.
- **FR-007 (validation strategy)**: After writing, the adapter MUST verify: the JSON
  is valid + schema-conformant (schema treated as UNCERTAIN -> conservative,
  well-known keys only), R1 (model-reference integrity) stays green, and a
  read->write->read round-trip is stable. A new `retail check` authoring-lint MUST
  cover the written PBIR (an allow-list-only, no-forbidden-key check -- the DL1
  analogue for report files).
- **FR-008-DEP (dependence on the generated theme)**: The adapter MUST consume a
  theme produced by `retail theme-gen` (Slice 1) as its formatting source of truth;
  it MUST NOT invent colors/fonts outside the supplied theme. (Named FR-008-DEP to
  avoid confusion with spec-001 FR-008.)
- **FR-009-BG (backgrounds as surface-2 output)**: The adapter MUST reference a
  committed surface-2 background asset (per `background-spec.yaml`) as a page
  background; it MUST NOT generate the image itself in this feature (image
  authoring stays an external/Claude-as-designer step) and MUST honor the surface-2
  purity rule (no data baked into the image reference).
- **FR-010 (the owner-approval gate; Principle V)**: The adapter MUST refuse to
  write any PBIR until a ratified ADR lifting spec-001 FR-008/FR-009 *for this
  adapter* is recorded. The agent MUST NOT self-grant this. This spec drafts the
  ADR; a named owner ratifies it before implementation.
- **FR-011 (readiness honesty)**: The adapter MUST NOT self-grant any readiness
  `pass` or emit a numeric confidence/health/maturity score (hard rule #9). A
  written report's `dashboard_ready`/design-review sign-off remains the verb owner's
  recorded human decision.

### Key Entities

- **Theme (input)**: the `retail theme-gen` output triplet (tokens + theme JSON +
  spec) -- the formatting source of truth.
- **PBIR report (target)**: the committed `powerbi/*.Report/` tree (page JSON +
  per-visual `visual.json`) the adapter writes into.
- **Allow-list (constant)**: the declared set of editable formatting properties --
  a committed, guard-tested constant.
- **Background asset (input, US2)**: a committed surface-2 image referenced by path.
- **The ADR (gate)**: the ratified record lifting FR-008/009 for this adapter.

## Success Criteria *(mandatory)*

- **SC-001**: Running the adapter on a fixture PBIR report + a generated theme
  produces a valid, round-trip-stable report whose visuals carry the theme's
  formatting -- verified by an automated test, not a human eyeball.
- **SC-002**: Re-running the adapter on an already-styled report produces a ZERO
  diff (idempotent / deterministic).
- **SC-003**: An edit outside the allow-list, a path-traversal input, or an
  invalid-PBIR result each causes the adapter to write nothing and exit non-zero.
- **SC-004**: With no ratified FR-008/009 lift recorded, the adapter refuses to
  write (the gate holds) -- verified by a test.
- **SC-005**: The new authoring-lint (`retail check`) flags a hand-planted
  forbidden key in a written PBIR; a clean written report passes.
- **SC-006**: `retail check` R1 stays green on every adapter-written report.

## Assumptions

- The theme generator (Slice 1) is merged and is the formatting source of truth.
- The PBIR/PBIP JSON schema is treated as UNCERTAIN -> the adapter writes only
  conservative, well-known keys and validates by round-trip, not by claiming schema
  completeness.
- Background IMAGE authoring stays external (surface-2 / Claude-as-designer); this
  adapter only references a committed asset.
- The owner will ratify (or reject) the draft ADR before any implementation task
  runs; no PBIR is written before that.
- Live publish to a workspace remains F016's deferred territory -- out of scope.
