# Readiness View -- the seven-stage stage-centric lens

> On-disk spec: `specs/020-readiness-viewer/`  Roadmap feature: F026 (the roadmap
> F-number is authoritative when the two disagree).
>
> GENERIC template. The stage-lens sibling of `data-quality-control-room.md`: ONE
> stage-centric view over each item's readiness, rendered by the `readiness-viewer`
> skill from evidence that ALREADY EXISTS per item in
> `mappings/<item>/readiness-status.yaml`. It runs NO new check and edits NO
> per-item file (read-only). See `../docs/readiness/readiness-model.md`.
>
> **Renders, never re-derives.** Every matrix cell is a per-stage `status` copied
> VERBATIM from the source; every evidence row is a reference copied as-recorded;
> every approval is copied {stage, owner, date}. Never an adjective, never an
> invented status, never a synthesized line/anchor.
>
> **No fake confidence.** There is NO health / confidence / percent-ready score
> column, and a filled copy MUST NOT add one (hard rule #9). Readiness is the four
> explicit statuses + evidence + approvals.
>
> **Same inputs as F012, different lens.** F012 owns findings + blockers
> (worst-first); this view owns the stage matrix + evidence references + approvals
> timeline. `next_action` is SHARED with F012, NOT a delta.
>
> **Generic, not C086.** Placeholder rows only; C086 / retail_store_sales are cited
> filled instances, never inlined. ASCII only, UTF-8 no BOM; the stage spine is
> written Source Ready -> ... -> Publish Ready (ASCII `->`).

## 1. The seven-stage status matrix (one row per item)

One row per discovered `mappings/<item>/readiness-status.yaml`. Each cell is the
recorded `stages.<stage>.status` (one of `not_started` / `blocked` / `warning` /
`pass`), copied VERBATIM -- never recomputed or upgraded. The `current_stage` is
marked `(*)`. The single `next_action` is shown. An item with no readiness file is
shown "no readiness file" and gets NO invented stage statuses.

| Item | Source Ready | Mapping Ready | Silver Ready | Gold Ready | Semantic Model Ready | Dashboard Ready | Publish Ready | Current stage | Next action |
|------|--------------|---------------|--------------|------------|----------------------|-----------------|---------------|---------------|-------------|
| `<item-a>` | `pass` | `pass` | `pass` | `blocked` | `not_started` | `not_started` | `not_started` | `gold_ready (*)` | `<single next action>` |
| `<item-b>` | `pass` | `warning` | `not_started` | `not_started` | `not_started` | `not_started` | `not_started` | `mapping_ready (*)` | `<next action>` |
| `<item-c>` | -- | -- | -- | -- | -- | -- | -- | no readiness file | -- |

- Statuses are copied from `stages.<stage>.status`; `(*)` marks `current_stage`.
- A malformed / partial file is shown "readiness file incomplete: `<file>`" for the
  affected fields -- the missing statuses are NOT guessed.

## 2. Evidence references per stage (navigable, missing shown as missing)

For a chosen item, each stage's `evidence[]` rendered as the committed reference(s)
backing that stage -- copied VERBATIM. A bare path stays a bare path; a line/section
anchor is rendered only if the source records it (NEVER synthesized). Empty
`evidence[]` -> "evidence missing" with the expected artifact named; a `pass` stage
with empty `evidence[]` -> the explicit "evidence missing for a pass stage" flag; a
referenced file absent on disk -> "referenced file not found" (the entry is kept,
never replaced).

| Stage | Status | Evidence reference (verbatim) | Flag |
|-------|--------|-------------------------------|------|
| Source Ready | `pass` | `mappings/<item>/<evidence-file>` | -- |
| Mapping Ready | `pass` | `mappings/<item>/<evidence-file>` | -- |
| Silver Ready | `pass` | `<committed path as recorded>` | -- |
| Gold Ready | `pass` | -- | evidence missing for a pass stage |
| Semantic Model Ready | `not_started` | -- | evidence missing (expected: `<artifact>`) |
| Dashboard Ready | `not_started` | -- | -- |
| Publish Ready | `not_started` | `<recorded path>` | referenced file not found |

- Evidence is rendered exactly as `stages.<stage>.evidence[]` records it; nothing
  fabricated or filled in.

## 3. The approvals timeline (chronological, gaps shown as gaps)

For a chosen item, each `approvals[]` entry as {stage/gate, named owner, date} in
date order, copied VERBATIM. A `pass` gate is flagged "approval not recorded" ONLY
when that stage's `docs/readiness/<stage>-ready.md` "Required owner / approval"
declares an approver IS required AND no matching `approvals[]` entry exists. Where
the stage doc declares NO required approver, an empty `approvals[]` is NORMAL and is
NOT flagged. The viewer never establishes, infers, or back-fills an approval.

| When | Stage / gate approved | Named owner | Flag |
|------|-----------------------|-------------|------|
| `<YYYY-MM-DD>` | `mapping_ready` | `<analyst>` | -- |
| `<YYYY-MM-DD>` | `semantic_model_ready` | `<governance>` | -- |
| -- | `publish_ready` | -- | approval not recorded (stage doc requires an approver) |
| -- | `gold_ready` | -- | normal -- stage doc requires no approver (not flagged) |

## 4. Conflicts surfaced (never resolved)

Surface, never reconcile. The viewer is a lens, not an arbiter (Principle V).

| Item | Conflict | What was observed | Action |
|------|----------|-------------------|--------|
| `<item>` | current-stage vs status | `current_stage: <stage>` but an earlier stage recorded `blocked` | surfaced as a flag; NOT resolved by picking one |
| `<item>` | pass without evidence | a `pass` stage with empty `evidence[]` | surfaced "pass without evidence"; never hidden |
| `<item>` | stray approval | an `approvals[]` entry references a `not_started` stage | surfaced as a conflict; the approval is not deleted |

## Notes

- This view is read-only and point-in-time. A historical trend over time is the
  Reconciliation Ledger's job (F015), not this view.
- The viewer renders recorded approvals and FLAGS missing required ones; recording
  an approval, moving a stage to `pass`, and clearing a blocker all remain Core
  Authority actions by the named owner (Principle V) -- never the viewer.
- A request for one readiness / confidence / percent-ready score is DECLINED, citing
  readiness-model "No fake confidence"; the four explicit statuses are returned.
