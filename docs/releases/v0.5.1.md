# Seshat BI -- v0.5.1 patch release candidate

This patch candidate records the changes merged after the frozen v0.5.0 release note. It does not rewrite the historical claims captured in `docs/releases/v0.5.md`.

## What this patch adds

- **`seshat scaffold-source <table>`** (issue #339, PR #342): writes the three Stage-1 blank templates (`source-profile.md`, `readiness-status.yaml`, `source-map.yaml`) into `mappings/<table>/` from bundled package data, so a pip-only workspace can produce the first Source-Ready artifact without the development repository. The three templates now ship as wheel package data and are required by the release-artifact gate; the materialized `readiness-status.yaml` carries a truthful `current_stage: source_ready` with empty evidence/approvals so a committed scaffold passes the RS1 governance gate as an honest unstarted journey.

## What this patch fixes

- **`validate` / `drift` / `value-check` now honor the workspace `.env`** (issue #340, PR #344): the live commands read `os.environ` only -- for both engine selection and connection resolution -- so a user who put the `ANALYTICS_DB_*` credentials in the gitignored `.env`, exactly as the error text, `.env.example`, and README instruct, got "no database connection configured" or silently the wrong engine. A new `applied_dotenv` context manager applies `.env` for the command body (real environment variables still win, and the process is restored on exit) without adding a `python-dotenv` dependency. Malformed or invalid connection settings now fail clean, without a traceback. `drift`'s live PostgreSQL leg remains gated on an explicit `--dsn` (it does not resolve `ANALYTICS_DB_*` from `.env` on Postgres); the `.env` fix covers engine selection, the non-Postgres path, and the connection-env parity for `drift`, and is complete for `validate` and `value-check`. Closing that last `drift` gap is a tracked follow-up.
- **`dbt plan` no longer swallows the underlying dbt parse error** (issue #341, PR #343): a failed non-database PARSE runs under `--log-format json`, so the Compilation Error landed on stdout while stderr was empty and `_successful` emitted a bare `DBT_ARTIFACT_INTEGRITY:` with nothing after the colon. It now surfaces the most informative available text so the real parse error reaches the operator.

## Honest boundaries

- The scaffolded Stage-1 artifacts are blank templates for the operator to fill; they carry no fabricated evidence, approvals, or readiness confidence.
- Live `validate` / `drift` / `value-check` still require the optional `db` extra and a real DSN; without them the surfaces report their deferred state and never fake a pass.
- A residual time-of-check-to-time-of-use race in the scaffolder's per-file write (a concurrent hostile process inside the workspace) is tracked as a scoped follow-up (#345); every deterministic path-safety case is closed.
- This candidate does not itself create a tag, publish to PyPI, create a GitHub Release, or submit to public Claude, Codex, or OpenAI catalogs.
