# Seshat BI -- v0.1 release snapshot

A detailed snapshot of what `main` is at v0.1. This is a
**downloadable / useable kit snapshot, not a fully packaged platform.** You can
clone it, verify it locally, and run the agent workflow against your own retail
source. It is not a published package or a managed service.

For the top-level summary, see [`RELEASE_NOTES.md`](../../RELEASE_NOTES.md).

## Product identity

Seshat BI is an **agent-first Retail BI readiness system**. It exists to answer
one question safely: *is this retail source ready to become trusted Power BI
analytics?* Readiness is `status` + `evidence` + `blocking_reasons`, never a
faked confidence score.

- **Power BI is a reporting target, not the source of truth.** The medallion
  warehouse (`gold`) is the source of truth; Power BI reads `gold` only.
- The current CLI / package alias is **`retail`** (`pyproject.toml`).
- The product is the **readiness system** -- gates, contracts, examples, and the
  agent workflow -- not Power BI automation.

## Who this release is for

- An **AI agent** (or a BI developer) that needs a disciplined, gate-by-gate way
  to take a raw retail source toward governed Power BI analytics.
- A **new user** evaluating the kit who wants to read the repo, run the static
  checks locally, and follow two worked examples end to end.

## What problem it solves

Retail BI usually skips steps: data lands straight in a dashboard, metrics are
invented in DAX, and nobody can say what a number means or whether it was
validated. Seshat BI makes the discipline structural -- no source goes directly to
silver, no gold reaches Power BI before validation, and no dashboard is designed
before its metrics are defined. Each transition is a recorded, reviewable gate.

## What is shipped

- The **seven-stage readiness spine**: Source Ready, Mapping Ready, Silver Ready,
  Gold Ready, Semantic Model Ready, Dashboard Ready, Publish Ready.
- The **source-mapping gate** -- `source-map.yaml` must be reviewed before any
  `silver.*` SQL is written.
- **Medallion warehouse conventions** under `warehouse/` (`bronze` / `silver` /
  `gold` + numbered migrations).
- The **`retail` CLI**: `retail check` (static gate), `retail validate` (live),
  `retail semantic-check` (contract drift), `retail value-check` (value proxy),
  and `retail generate` (DAX from an approved contract).
- The **SQL**, **DAX**, and **Python** knowledge layers.
- Two worked examples:
  - [`docs/worked-examples/c086-pharmacy.md`](../worked-examples/c086-pharmacy.md)
    -- the first filled example (proof of the pattern, not the universal schema).
  - [`docs/worked-examples/retail-store-sales.md`](../worked-examples/retail-store-sales.md)
    -- the second domain; the first to traverse the full spine to Dashboard
    Ready, with Publish Ready honestly at `warning`.
- **Power BI / PBIP governance and handoff** conventions (plain-text TMDL/PBIR;
  handoff packs; gold-only reads).

## What is deliberately deferred

These are gated or deferred by design, not missing through neglect:

- **F016 -- Power BI execution adapter.** Execution-only; deferred and gated; not
  startable before `semantic_model_ready` is `pass`.
- **One-click dashboard generation.** Not a goal; dashboards are designed from
  approved metric contracts.
- **Automated ingestion / live DB provisioning.** Manual load now.
- **Fully automated mapping approval.** The mapping gate requires human review.

## What a successful first local run looks like

With Python 3.13 and a clean checkout:

```bash
pip install -e ".[dev]"
ruff format --check src tests
ruff check src tests
pytest -m unit
retail check
retail semantic-check --repo .
```

A successful first run means: the static checks report clean, the unit suite
passes, and `retail check` / `retail semantic-check` exit cleanly. Live
validation (`retail validate`, `retail value-check`) only runs when a database is
configured -- without one it is **pending, not passed**. See
[`docs/quality/local-verification.md`](../quality/local-verification.md).
