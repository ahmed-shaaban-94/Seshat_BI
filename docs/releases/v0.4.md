# Seshat BI -- v0.4 release candidate

This release candidate records the repository state prepared from `main`
after the governed dbt and Dagster adapter activations (PRs #299-#303,
2026-07-16/17). It is a repository candidate only: it does not prove PyPI,
GitHub Release, Claude catalog, Codex catalog, or OpenAI public-plugin
availability. Each of those actions requires its own named-owner approval and
external evidence.

## Product identity

Seshat BI remains an **agent-first BI readiness system**. It requires evidence
and named-human approvals at readiness gates; it does not turn an agent
recommendation, a version number, or a release-candidate audit into an
approval.

## What this candidate adds since v0.3.1

- **Governed dbt transformation MVP** (spec 133, PR #299): an exact optional
  `dbt-core==1.12.0` + `dbt-postgres==1.10.2` runtime (`seshat-bi[dbt]`), a
  tracked eight-model `retail_store_sales` shadow graph with 24 selected
  tests, Mapping Ready and source-map citation validation, immutable accepted
  plans, invocation locks, redacted subprocess handling, normalized parity
  evidence, the `seshat dbt` doctor/validate/plan/build/test/inspect-run
  command family, and the shared `dbt-workflows` Claude/Codex skill.
  **Honest label**: static parse/list and artifact compatibility are locally
  verified; live compile/build/test/parity and the named-owner compatibility
  attestation remain `[PENDING LIVE PROFILE]`. Migrations remain the default
  build path.
- **Dagster orchestration MVP** (spec 134 / F030, PR #300): a real
  `orchestration/dagster/` runtime project (`tower_bi_orchestration`,
  `dagster==1.13.14` + `dagster-dbt==0.29.14` pinned together, own
  environment) running the full 11-asset medallion graph behind every gate --
  fail-closed STOP edges, read-only HUMAN-SEAM approval reads, a publish wall
  that only TRIGGERS the publish feature and fails closed while it is absent.
  The `seshat dagster doctor|run|evidence` command family (exit codes 0..4),
  derived schema-validated run evidence, and a definitions-load CI smoke.
  **Honest label**: locally verified; the one daily schedule and one
  raw-landing sensor ship STOPPED, and starting either is a named-human
  action. The `dagster-dbt` engine seam remains documented, not activated.
- **Canonical public command surface and normalized command names** (PR #297):
  `distribution/public-command-surface.yaml` is the single authority for what
  the generated agent bundles advertise, reconciled by contract drift gates;
  core readiness commands use bare verb names with the four v0.2.0-accepted
  `seshat-*` names kept as deprecated aliases for one release cycle.
- **Eight Power BI plugin commands and the shared `powerbi-workflows` skill**
  (PR #297), generated through the existing allowlist/exporter for both
  bundles.
- **`dagster-workflows` public skill and Dagster surface parity** (PR #303):
  the governed Dagster workflow ships as a dedicated shared skill in BOTH the
  Claude and Codex bundles, routed from the three `dagster-*` commands and the
  `seshat-bi` router, with the Dagster capability entry enriched to the dbt
  reference standard.
- **Bundle provenance hardened against squash-merges** (PRs #301, #302): the
  everyday export/regeneration posture validates the manifest version claim
  against HEAD's canonical `pyproject.toml` when history rewriting orphans the
  recorded generation commit; the coordinated-release audit keeps strict
  ancestry.

## What this candidate does NOT claim

- No live dbt compile/build/test/parity evidence exists yet; every dbt
  live-path claim is `[PENDING LIVE PROFILE]`.
- Dagster automations are STOPPED; no unattended run has been scheduled.
- The `dagster-dbt` engine seam (silver/gold via dbt assets inside the
  Dagster graph) is documentation only and requires its own spec, gate
  review, and named-owner approval before any code activates it.
- No publication action (PyPI, GitHub Release, Claude/OpenAI catalogs) is
  performed or implied by this candidate.
