# Seshat BI -- v0.5.3 patch release candidate

This patch candidate records the changes merged to `main` after the frozen
v0.5.2 release note. It does not rewrite the historical claims captured in
`docs/releases/v0.5.md`, `docs/releases/v0.5.1.md`, or `docs/releases/v0.5.2.md`.

## What this patch adds

- **CLI identity follows the invoked command + `--version`** (issue #378): the
  parser's `prog` name now derives from `argv[0]`'s basename (falling back to
  `retail`), so `seshat --help` / `seshat doctor` print their own identity
  instead of a hardcoded `retail`. A new top-level `--version` prints `<prog>
  <version>` from installed package metadata, with a `0+unknown` fallback for
  an uninstalled source tree. `main()`'s existing SystemExit-catching contract
  is unchanged.

## What this patch fixes

- **A raw traceback on broken/unlaunchable git** (issue #394): `check` and
  `doctor` now catch `(OSError, RuntimeError)` around their `build_context`
  call and emit a clean stderr error + exit 1, instead of propagating a raw
  traceback when git is missing or fails non-zero/non-128. Scoped to the two
  unguarded consumers; git exit-128 (not-a-repo / no-HEAD) is still tolerated
  inside `_git_ls_files`, so a non-git workspace is unaffected.
- **Three more DSN shapes under-scrubbed by `redaction_core`** (issues #392,
  #393): a mixed-case hostname, URI query-param credentials
  (`?user=..&password=..`), and libpq keyword conninfo strings
  (`host=.. user=.. password=..`) are now decomposed and scrubbed at the one
  shared core, so every boundary redactor benefits. The P2 repo-root probe
  switches from a `Path.samefile` comparison (which a Cygwin/MSYS git's POSIX
  toplevel could fail) to `git rev-parse --show-prefix`.
- **A duplicated, weaker DSN redactor and a P2 false-error in non-git
  workspaces** (issues #384, #385): `cli._redact_dsn` is reimplemented on top
  of `seshat.redaction_core`, so the DB name and percent-decoded credential
  forms are scrubbed too and the hand-rolled copy can no longer drift from the
  hardened core. A fully non-git workspace is now treated like the existing
  git-init'd-but-no-HEAD case, so P2 no longer errors with "could not read
  commit range" there.
- **`doctor` disagreeing with `check` on a foreign (non-bootstrapped) repo**
  (issue #377): `doctor`'s KIT_SELF aggregation is now gated on
  `is_bootstrapped(ctx.repo_root)`, matching `check`'s existing skip, so a
  client's downloaded-into workspace is not warned about kit-internal
  manifests it was never given.
- **`scaffold-source` output inconsistent with its own templates** (issues
  #374, #380): the written `readiness-status.yaml` next_action now matches the
  actual `source_ready` stage (previously it carried a mapping-stage action and
  tripped `seshat next`'s own disagreement caveat), and all five sister
  artifacts declared in the table's `source-map.yaml` are materialized
  (`assumptions.md` and `reconciliation-report.md` were previously missing).
- **`demo run` / `demo load` DSN and live-mode drift** (issues #375, #376,
  #379): `demo run`'s live mode is now decided by an actual
  `probe_reachable(dsn)` connect result instead of `bool(dsn)`, and both verbs
  resolve the DSN through one shared `demo/_dsn.py` helper (workspace `.env`
  included). A live-leg connect failure is now redacted through `_redact_dsn`
  before being reported, closing a path where psycopg2's reformatted error
  text could surface host/user/password in an unhandled traceback.
- **`seshat init` crashing on a fresh non-git workspace** (issues #370, #371,
  #372): `.seshat/kit-source.yaml` now ships as a bundled template (seeded by
  `kit_init.bootstrap()`), `gitutil.git_check_ignore` treats git exit-128 as a
  clean "not ignored" (matching its sibling wrapper), and P1 checks the
  filesystem instead of the git-tracked set (empty in a non-repo) so it no
  longer falsely reports on-disk files as missing.

## What this patch changes internally

- **`retail check` renamed to `seshat check` in user-facing surfaces**: the
  rebrand is documentation/output-text only in this patch; no CLI behavior
  changes.

## Honest boundaries

- The redaction fixes narrow real leak paths under the shapes investigated;
  they do not claim to redact every conceivable driver-error reformatting, and
  the scrubbers fail closed (drop to a generic boundary error) when they
  cannot guarantee a clean string.
- A second `#393` item -- distinguishing "broken git" from "not a repo" with a
  proper P2 ERROR rather than a silent skip -- is intentionally deferred; it
  needs a `gitutil` exit-code dispatch refactor that exceeds this patch's scope.
- This candidate does not itself create a tag, publish to PyPI, create a
  GitHub Release, or submit to public Claude, Codex, or OpenAI catalogs.
  External marketplace publication remains a separate owner action.
