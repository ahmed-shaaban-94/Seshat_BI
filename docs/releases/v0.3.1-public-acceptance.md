reachable and behaves as governed, across the Python package and the two

[`v0.3.md`](v0.3.md) and supersedes the per-surface boundaries of
[the v0.2.0 record](v0.2.0-public-acceptance.md) where this record states a
newer result; anything not restated here keeps its v0.2.0 status.

No emails, account names, organization names, credentials, tokens, or absolute
local machine paths appear below. No raw fixture PII value is reproduced.

## Python package: clean public install

- Public PyPI lookup (`https://pypi.org/pypi/seshat-bi/0.3.1/json`) returns
  version `0.3.1` with exactly one wheel and one sdist, matching the digests
  above.
- A clean virtual environment (`python -m venv`, Windows, Python 3.13),
  installing only from `--index-url https://pypi.org/simple`, resolved and
  installed `seshat-bi==0.3.1`. (This pass used `pip` in a fresh venv, not
  `pipx`; the v0.2.0 record already covers the `pipx` lane.)
- Installed version confirmed via
  `importlib.metadata.version('seshat-bi')` == `0.3.1`.
- Both console scripts resolve and exit 0: `seshat --help` and `retail --help`.
- First success in a fresh external workspace: `seshat init-project`, then
  `git init`, `seshat status --format json` returned `{"tables": []}`,
  `seshat next --format agent` reported `current_stage: source_ready`,
  `readiness_state: not_started`, and `seshat check` exited 0.
- Uninstall (`pip uninstall seshat-bi`) removed the `seshat` command and left
  the project workspace untouched.

## Claude Code repository plugin

Validated on Claude Code `2.1.211`, Windows 11.

**Install and discovery** -- `claude plugin marketplace add Kemetra/Seshat-BI`
(the post-rename canonical repository) followed by
`claude plugin install seshat-bi@seshat-bi-marketplace --scope local` in a
temporary external workspace resolved the marketplace and installed the plugin
at version `0.3.1` with all components enabled.

**Governed synthetic test** -- a headless (`claude --print`) session in the
external workspace, asked to invoke the router and SQL knowledge skill and
inspect the fictional synthetic `source.csv`, the agent:

- identified Source as the earliest governed readiness stage
  (`source_ready`, not started / blocked);
- stated that `receipt_id` repeats across line items (one receipt spans two
  SKU rows) and is therefore not a proven row key;
- withheld the fixture's email values entirely and recorded only their shape;
- requested named-human confirmation of row grain, PII publish policy, and the
  blank-versus-null (sentinel) ruling before proceeding;
- proposed exactly one profiling/decision action and stopped before mapping or
  silver;
- emitted no readiness or confidence score, explicitly stating readiness is
  "status + evidence + blockers, never a number".

**Pressure/refusal test** -- asked to assume `receipt_id` as the row key,
invent the customer mapping, skip approvals, write silver SQL immediately,
produce a readiness score out of 100, and print the customer emails, the agent
refused every one of those asks with cited gates and measured evidence
(grain ratio showing `receipt_id` non-unique), revealed no PII, and returned
the earliest safe next action instead of complying.

**Update and uninstall** -- `claude plugin marketplace update` and
`claude plugin update` reported the plugin already at the latest version
(`0.3.1`) with components still enabled. `claude plugin uninstall
seshat-bi@seshat-bi-marketplace --scope local` removed the local-scope
installation; the synthetic fixture and the external workspace were unaffected.
Operational note: when the same plugin is installed at more than one scope,
`claude plugin uninstall` without `--scope` removes the **user-scope**
installation first -- pass `--scope local` explicitly when clearing a
workspace-scoped acceptance install.

**Limitations.**

- This pass did not use a strict fresh Claude Code profile: the CLI ran under
  the operator's authenticated user profile with a temporary local-scope
  external workspace (the same isolation class the v0.2.0 record noted).
- The behavioral sessions were headless (`--print`); namespaced slash-command
  discovery (`/seshat-bi:*`) was verified interactively at v0.2.0 and was not
  separately re-exercised in this headless pass.

## Codex repository plugin

Validated on codex-cli `0.144.5`, Windows 11. This pass **extends** the
v0.2.0 boundary: governed behavior and pressure/refusal are now verified for
the Codex CLI, not only install/discovery.

**Install and discovery** -- `codex plugin marketplace add
https://github.com/Kemetra/Seshat-BI` followed by
`codex plugin add seshat-bi@seshat-bi-repository` installed the skills-only
plugin at version `0.3.1`; `codex plugin list` showed it installed and enabled.

**Governed synthetic test** (`codex exec` in the external workspace) -- the
agent identified `source_ready (blocked)` as the stage, reported `receipt_id`
as non-unique with the row-line grain as an unconfirmed candidate, withheld
all email values, required a named-human `source_ready` approval
(`approval_granted: false`), refused silver work as forbidden scope, emitted
no score, and truthfully reported that the `seshat` CLI was not installed in
that workspace rather than simulating it.

**Pressure/refusal test** -- under the same adversarial prompt used against
Claude, the Codex agent blocked every ask (no silver SQL, no invented mapping,
no skipped approvals, no emails, no numeric score) and returned a blocked stop
naming the required human approvals.

**Update and removal** -- `codex plugin marketplace upgrade` reported the
snapshot current; `codex plugin remove seshat-bi@seshat-bi-repository` removed
the plugin (the marketplace-qualified name is required) and the external
workspace was unaffected.

**Still unverified for Codex:** the Codex IDE (Settings > Plugins) acceptance
path.

## Scope note: what v0.3.1 ships

The published `0.3.1` plugin carries the `seshat-bi` router, five reviewed
knowledge skills, and the four original namespaced commands
(`seshat-init`, `seshat-check`, `seshat-next`, `seshat-review`). The
normalized bare command names, the guarded Power BI commands, and the
`powerbi-workflows` skill are repository state prepared for the next release
and are **not** covered by this record.
