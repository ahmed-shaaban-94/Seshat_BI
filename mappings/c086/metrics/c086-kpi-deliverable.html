01

Executive read

El Ezaby C086 took EGP 38.80M net across three years (2023–2025), with a pharmacy-typical prescription tilt: RX drives 48.16% of net sales, OTC another 29.00%, and nutraceuticals 13.33% — the top three divisions alone are 90.5% of the book. Product concentration is severe: the top 20% of products generate 88.30% of net sales, a textbook Pareto that says shelf, stock and promotion decisions should be ruthlessly SKU-weighted. Growth stalled after a strong 2024 — net sales rose +12.9% in 2024 then slipped −3.9% in 2025 — so the story is a plateau, not a climb. Baskets are small and frequent: AOV EGP 378.28 across 102,579 transactions at 2.79 units each, and roughly 72% of sales fall on weekdays. Returns are contained — a 7.46% value return rate (4.29% of lines) — and returns already net into the headline via the negative-sign convention, so the EGP 38.80M figure is net, not gross. One structural caveat colours the whole customer view: 84,770 lines collapse into a single anonymous WALK_IN identity (23.63% of net sales), which makes whole-base retention impossible to compute honestly — flagged below, not fabricated.

03

DAX measure sheet

Paste-ready and grouped by display folder. All references use the deployed model's quoted table names 'gold fct_sales' / 'gold dim_…' — the reviewers' single most-repeated note was that the shorthand fct_sales[…] will not resolve against a table whose real name contains a space. Author the two base measures NetSales and TotalUnits first; the rest reference them.

01

Sales

4 measures
NetSalesbase measure

Headline figure. A plain SUM is already net of returns because return lines carry negative gross_sales — do not add a filter.

// Sales — the headline; already net of returns via signed rows
NetSales = SUM ( 'gold fct_sales'[gross_sales] )
GrossSales

Kept lines only. is_return is a physical boolean, so the predicate is exact.

GrossSales =
CALCULATE ( [NetSales], 'gold fct_sales'[is_return] = FALSE () )
Returns

Negative by design — never wrap in ABS here. GrossSales + Returns = NetSales, penny-exact.

Returns =
CALCULATE ( [NetSales], 'gold fct_sales'[is_return] = TRUE () )
TotalUnitsbase measure

Net units — same sign convention as NetSales. Report to 3 dp (NUMERIC(18,3)); do not round to integer.

// quantity is signed; plain SUM nets returned units automatically
TotalUnits = SUM ( 'gold fct_sales'[quantity] )
02

Returns

2 measures
ReturnValueRate

Value-weighted return rate. ABS is applied here (not in Returns) to turn the negative drag into a positive rate. Denominator is Gross, not Net.

ReturnValueRate =
DIVIDE ( ABS ( [Returns] ), [GrossSales] )
LineReturnRate

Count-based sibling of ReturnValueRate. Usage caveat: do not slice by is_return — the CALCULATE predicate overrides it.

LineReturnRate =
DIVIDE (
    CALCULATE ( COUNTROWS ( 'gold fct_sales' ), 'gold fct_sales'[is_return] = TRUE () ),
    COUNTROWS ( 'gold fct_sales' )
)

Basket / Transaction

5 measures
Transactions

Distinct invoices (degenerate id on the fact). Non-additive across line-grain attributes — a mixed invoice counts in every product slice it touches.

Transactions = DISTINCTCOUNT ( 'gold fct_sales'[invoice] )
SaleTransactionswas BROKEN

Invoices with ≥1 non-return line; all-return baskets drop out. Corrected: the reviewer's original used the unquoted fct_sales which cannot resolve against the space-named table.

SaleTransactions =
CALCULATE (
    DISTINCTCOUNT ( 'gold fct_sales'[invoice] ),
    'gold fct_sales'[is_return] = FALSE ()
)
AverageOrderValue

Net-per-basket. Denominator is ALL transactions (not sale-only) or the value drifts.

AverageOrderValue = DIVIDE ( [NetSales], [Transactions] )
UnitsPerBasket

Net units per transaction — fractional result is correct (NUMERIC(18,3)).

UnitsPerBasket = DIVIDE ( [TotalUnits], [Transactions] )
AverageLineValue

Net money per line across all 246,916 fact rows (returns included in the denominator).

AverageLineValue = DIVIDE ( [NetSales], COUNTROWS ( 'gold fct_sales' ) )

WalkInSalesShare

ALL('gold dim_customer') keeps the denominator at full net even under a customer slicer — a true "fraction of everything". Intended as a card, not a per-customer column.

    DIVIDE ( WalkInNet, TotalNet )

04

Feasibility ledger

What the data cannot support, and what ships with a recorded caveat. The discipline here is the deliverable: a blocked KPI is a refusal to fabricate, not a gap in the work.

Deliberately excluded — the data cannot answer honestly

Margin · GP% · Cost · Discount · Tax

The fact carries only two measures: gross_sales and quantity. There is no cost, landed cost, discount, or tax column anywhere in gold. Any margin, gross-profit, markup, or discount-rate KPI is impossible and was never authored — inventing one would require a cost column that does not exist.

CustomerRepeatRate — whole-base retention

84,770 anonymous lines collapse into a single WALK_IN identity, so repeat-purchase is undefined for the anonymous majority. A whole-base retention number would either treat WALK_IN as one always-repeating customer or silently drop the majority — both fabricate. Status: blocked (Principle V). A named-only variant is definable but has no ground-truth anchor; ship it separately at warning, verified live.

Ships with a caveat — warnings the owner must accept before promotion

  • Table-name resolution (whole batch) — every measure references 'gold fct_sales'/'gold dim_…' with the space-prefixed quoted name. The shorthand fct_sales[…] in the original proposals does not resolve; all DAX above is already corrected.
  • Base measures unbuilt — the C086 semantic model currently has zero measures. NetSales and TotalUnits must be authored first; every composed measure depends on them.
  • SaleTransactions, SalesMixPctByDivision, SalesPerProduct — reviewer verdict BROKEN, now corrected. SalesPerProduct also needed contract fixes: bind to gross_sales (not the fabricated sales_amount), and use the live distinct-sold product count, not the 9,669 catalog size.
  • SalesByBranch — 2-branch thinness — one real store plus the −1 unknown member; a cross-branch visual is effectively single-valued. Kept for multi-store future-proofing; owner confirms it is worth surfacing.
  • SalesPerSalesperson / SalesPerCounterAgent — row-set asymmetry — numerator includes unattributed (−1) sales that the denominator excludes, slightly inflating the per-agent average. Owner accepts or aligns predicates.
  • Derived / verify-live values — NamedCustomerSales (29.64M), WalkInSalesShare (23.63%), SalesPerCounterAgent (2.43M), and the rolling/MoM measures are internally consistent but derived by subtraction or lack a supplied anchor; confirm against a live query before sign-off.
  • ABCClassAShare — threshold ratification — the 20% ABC class-A cutoff is analyst-supplied business convention, not derived from data. Owner ratifies before promotion.
05

How to use this

The contract and the DAX are two separate deliverables, kept apart on purpose: contracts hold plain-language intent a non-technical owner can approve; the DAX is the implementation that lands in the PBIP model.

1
One contract file per KPI

Each KPI is a metric-contract.yaml instance at mappings/c086/metrics/<KpiName>.yaml, co-located with the table's mapping and readiness artifacts. Six already exist (ActiveCustomers, NamedCustomers, NamedCustomerSales, SalesPerNamedCustomer, WalkInSalesShare, CustomerRepeatRate); the rest follow the identical shape.

2
Group them with a KPI pack

A kpi-pack.yaml references the contracts by their stable PascalCase name, grouping the 33 into the eight families used above (Sales, Returns, Basket, Customer, Time Intelligence, Product, Staff, Segment).

3
Contracts stay plain-language intent

formula_intent is business meaning only — never DAX or SQL. binds_to is gold.* only. Every KPI here defaults to readiness.status: not_started; the agent never self-approves. A human owner promotes to pass with recorded evidence (owner name + date).

4
DAX goes into the PBIP model

Paste each measure from Section 03 into the c086 _sales.SemanticModel as a PascalCase measure, organized into the display folders shown. Author NetSales and TotalUnits first. This is the separate implementation stage — the define/check boundary — and is where the space-quoted table names matter.

5
Verify, then promote

Reconcile each measure against the expected value in the catalog. For the verify-live KPIs (SaleTransactions, MoM, rolling, top-product, per-product), run the live query first — do not stamp a derived or structural value as ground-truth-confirmed. Only then does the owner move a contract to pass.

Provenance. All expected values reconcile to the supplied live-verified ground truth for El Ezaby C086 (gold.fct_sales, 246,916 lines, 2023-01-01 → 2025-12-31). Verdicts and corrected DAX are taken from the adversarial review record. Currency is EGP (Egyptian pounds). No margin, cost, or discount metric appears because no such column exists in the gold model — this is a feasibility boundary, not an omission.
